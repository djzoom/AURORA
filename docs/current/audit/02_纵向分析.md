## 一、难度递进分析

### 1.1 整体难度弧线

Aurora 课程的难度曲线在宏观上呈现三段式结构：**平坦导入期**（L01–L31）→ **线性攀升期**（L32–L65）→ **陡增积分期**（L66–L91）→ **快速收束期**（L92–L99）。理想的课程难度曲线应为渐进 S 型，当前设计在两个关键位置出现了陡峭的"悬崖"，将对学生的学习连续性造成显著冲击。

```
难度
  A ┤                                              ╭─╮
  ┼ ┤                                         ╭───╯  │
  B ┤                          ╭──────────────╯      │
  ┼ ┤              ╭───────────╯                     ╰───╮
  C ┤──────────────╯                                     ╰──
    └──────────────┬──────────┬──────────┬──────────┬──────
                 L01       L32        L54         L66     L99
```

### 1.2 关键难度断崖

**断崖一：L37–L39（DFT 暴力实现 → Cooley-Tukey FFT，落差约 2 个认知等级）**

- L37 以 O(N²) 双循环实现 DFT，认知要求仅为"按公式写循环"
- L38 要求学生理解蝶形分治的偶奇拆分逻辑和旋转因子递推，认知要求跃升为"能推导递推式"
- L39 要求从零手写 Cooley-Tukey 递归实现并通过 < 1e-10 误差验证，且 L39 审计发现参考输出存在严重错误（cell 11 的"全实数"参考值与 `[1,2,3,4,3,2,1,0]` 信号实际 DFT 结果不符，4/8 个系数错误），学生在调试时无法区分"我的实现错了"和"参考答案错了"

该断崖的本质是：L37 仅需复现公式，L38–L39 需要内化一个全新的算法设计范式（分治 + 位运算）。课程在 L36 之前没有任何递归算法或分治思维的铺垫，跨度过大。

**断崖二：L54（从零手写自动微分，认知模式突变）**

- L53 的最后一课仍在谈 MFCC 可视化（直觉建立）
- L54 立即要求学生实现标量计算图的前向值 + 反向梯度，包括 `__init__`、`__add__`、`__mul__`、`__pow__` 和拓扑排序
- 审计发现 L54 的 `src/aurora/ml/` 目录根本不存在（该子包尚未创建），Aurora 连接断裂
- 没有任何从信号处理思维到计算图思维的过渡课节

该断崖是全课程最陡的认知跃升：学生从"音频工程师"模式突然切换到"编译器作者"模式，且无过渡。

**断崖三：L68–L69（CTC 对齐原理 → CTC 前向算法）**

- L68 的"单调路径"概念在标题中承诺但正文从未解释，学生带着概念空缺进入 L69
- L69 的前向 DP 要求掌握 log-domain 数值计算（logaddexp）、扩展标签构造和三路递推，是全课程数学密度最高的单节课
- L69 整体评级 C，且可运行性严重不足（`ctc_forward` 在构造 `lprime` 之后直接 `raise NotImplementedError`，Cell 6 和 Cell 8 无法运行）
- 验证仅检查 `np.isfinite` 和 `< 0`，任何返回 `-1.0` 的错误实现都能通过

### 1.3 过渡过于平缓的区域

L11–L21（线性代数 11 节）和 L27–L31（概率统计 5 节）的内部难度过于平坦。以线性代数为例：L09（向量加法）→ L10（点积）→ L11（范数）→ L12（矩阵乘法）→ ... → L17（特征分解）→ L18（可逆性与秩），每节的难度增量几乎相同，缺少在 L14（SVD）和 L17（特征分解）处的额外深度处理——这两个概念是面试中最高频的考核点，却在课程中与其他内容等价处理。

### 1.4 整合阶段的下行问题

L92–L99 的难度不升反降，从算法实现回落到系统设计和软技能。这在教学上是合理的（收束期），但 L92 的问题集中说明集成课并非容易：三个核心引用模块（`aurora.speech.whisper`、`aurora.rag.retriever`、`aurora.llm.agent`）在代码库中均不存在，"端到端流水线"的演示建立在虚假的模块连接之上，整体评级 C+。

## 二、前驱知识依赖检查

### 2.1 系统性超前引用问题

| 课节 | 引用概念/工具 | 实际引入课节 | 前驱差距 |
|------|-------------|------------|---------|
| L21 | MFCC 流水线描述 | L49–L50 | 28–29课 |
| L36 | periodic vs symmetric 窗 | 未显式讲授 | 终身缺席 |
| L54 | `src/aurora/ml/`（计算图库） | 未创建 | ∞ |
| L62 | PyTorch Dataset 机制（`__getitem__`） | L59–L61（前置 3 课） | 合理，但 L62 的 shape 约定与 L63 CNN 不兼容 |
| L66 | Levenshtein DP 实现 | L67（下一课才讲） | -1课（倒置！）|
| L71 | Beam Search（首次出现，无铺垫） | 本课首次 | 无前驱 |
| L83 | "nn.Linear 旁并联低秩矩阵" | L59–L61（PyTorch） | cell-18 混淆了技术栈 |
| L92 | `aurora.speech.whisper` | 未实现 | ∞（幻影模块）|

**最严重的超前引用：L66 要求学生实现 Levenshtein DP，但 L67 才讲解这个算法的推导。** 这是明确的教学顺序错误。学生在 L66 的练习中必须使用尚未学习的核心算法，而正确的做法是将 `compute_wer` 的实现练习后移至 L67 之后，或在 L66 中先给出完整 DP 推导（把 L67 的内容前置一个位置）。

### 2.2 L62 的深层前驱依赖问题

L62 要求学生实现 `KWSDataset.__getitem__`，调用 `aurora.audio.mel.mel_spectrogram` 并与 L63 的 CNN 对接。依赖链分析：

```
L32-L47（STFT+Mel实现）→ L59-L61（PyTorch基础）→ L62（Dataset）→ L63（CNN）→ L64（训练循环）
```

前驱知识本身是充分的，但 L62 的实现存在形状约定冲突：`mel_spectrogram` 返回 `(T, n_mels) = (63, 40)`（Aurora 约定：时间维在前），而 L63 CNN 期望 `(n_mels, T) = (40, T)`。这不是前驱知识缺失，而是跨课节接口约定不一致——是更难被发现的设计缺陷。

### 2.3 L68 对 L37–L39 FFT 深度的隐式依赖

L68 的 CTC 压缩率分析（声称 65-70%）以 V=30 的随机 logits 为例，但压缩率的直觉来自对词表大小的理解。正确推导需要理解概率计算（`(V-1)/V` 的去重存活率）而非 FFT 深度，因此 L37–L39 的深度不构成 L68 的前驱依赖。但 L68 对 **L27–L29（概率统计）** 的依赖是真实的——如果学生在概率模块的学习中未能掌握期望值计算，L68 的压缩率推导将无从理解。课程在 L28–L29 中虽然覆盖了描述性统计和分布，但没有显式演练"对 Categorical 分布的函数求期望"，导致 L68 的分析步骤对学生不透明。

## 三、跨课引用一致性

### 3.1 L 编号引用的准确性

基于抽样审计，跨课 L 编号引用准确性分布如下：

| 模块 | 样本课节 | 正确引用 | 错误/过时引用 | 典型错误 |
|------|---------|---------|------------|---------|
| Audio DSP | L36, L38, L39 | 高 | 1处 | L38 引用不存在的 `_fft_recursive()` |
| 深度学习 | L54, L62 | 中 | 2处 | L54 的 M2-A2 代号；`aurora/ml/` 不存在 |
| ASR | L66, L67, L68, L69, L71 | 中 | 2处 | L83 中 "L2" 错误引用；L79 中 "M4-MU3" 残留 |
| LLM/RAG | L83, L89 | 高 | 1处 | L89 的 L90 前向描述错误（工具调用 vs 对话 RAG）|
| 音乐 | L79 | 高 | 1处 | L79 "M4-MU3" 旧编号残留 |
| 集成 | L92, L95, L96 | 中 | 2处 | L96 Attention 引用 "L83-L84"（L84 是 LoRA）；L95 Whisper 与 Transformer 顺序矛盾 |

**关键发现：旧版 M 编号（M2-A2、M4-MU3）残留在至少 3 个课节中**，说明从旧的月度编号系统迁移到 L 编号时未全面替换。这些残留在学生角度是死链，会造成"课程已更新但文档未跟进"的困惑。

### 3.2 `src/aurora/` 路径引用的准确性

实测验证的关键路径引用：

**准确的引用（High Fidelity）：**
- L36 → `src/aurora/audio/windows.py`（hann/hamming/blackman 均已验证导出）
- L47 → `src/aurora/audio/mel.py`（mel_filterbank 存在且签名正确）
- L62 → `src/aurora/audio/mel.mel_spectrogram`（存在，但有形状约定争议）
- L89 → `aurora.llm.build_tfidf`、`cosine_retrieve`（在 `src/aurora/llm/retrieve.py` 中存在）

**断裂的引用（Fatal Breaks）：**
- L22 → `src/aurora/audio/grad_check.py`（文件不存在；audio 子包中只有 io.py、mel.py、mfcc.py、stft.py、transforms.py、windows.py）
- L38/L39 → `aurora.audio.transforms._fft_recursive()`（函数不存在；实际只有迭代式 `fft()`）
- L54 → `src/aurora/ml/`（整个子包不存在）
- L92 → `aurora.speech.whisper`、`aurora.rag.retriever`、`aurora.llm.agent`（均不存在）
- L79 → `aurora.music` 中的 `MusicEncoder`、`triplet_loss`、`nt_xent_loss`（均不在 `__init__` 中导出）

断裂引用中，L22 和 L38–L39 属于引用已实现但路径错误的情况（函数以其他名称存在），L54 和 L92 属于引用尚未实现的模块（计划功能）。两类问题的修复策略截然不同：前者只需更正路径，后者需要明确标注"计划中"或实际创建模块。

**内聚性最低：ASR (L66–L75)**，原因是教学顺序存在结构性错误（DP 练习先于 DP 讲解）、多处数学错误未修复、两节 C 级课降低模块整体可信度。

## 五、练习体系分析

### 5.1 TODO 练习分布

从抽样审计推断全课约有 180–220 个 TODO 练习单元，分布极不均匀：

- **Audio DSP（L32–L53）**：平均每课 2.5 个 TODO，验证断言明确（`np.allclose`，atol ≤ 1e-10）
- **深度学习（L54–L65）**：平均每课 2.0 个 TODO，但部分使用 `pass` 而非 `raise NotImplementedError`（L54、L66），导致学生在未完成时得到误导性错误
- **ASR（L66–L75）**：练习设计质量最差。L67 完全无需学生编写代码（答案已预填），L68 推理路线直接给出 Python 表达式（认知负荷接近零），L69 skeleton 中 `raise NotImplementedError` 替代了整个函数体（可运行性为零）
- **可视化课（L08、L19、L20、L26、L31 等）**：设计为"无编码任务"（参数实验），合理，但存在参数实验单元缺少可运行代码格的问题（L21 cell-19 只有文字描述）

### 5.2 练习设计质量分级

**A 级练习设计（少数亮点）：**
- L38 `butterfly()` 函数：正确使用 `raise NotImplementedError`（虽用 `...` 代替，略次）；验证包含计数实验（理论乘法次数 vs 实测），超越了一般的形状/数值验证
- L50 MFCC 流水线：要求学生逐步实现并在每个中间步骤验证 shape，提供了可以支撑"口述推导"的完整代码路径
- L79 Triplet Loss：提供闭式期望值作为验证目标（`sqrt(8)*10 + 0.2 ≈ 28.48`），让学生有数值锚点

**D 级练习设计（需优先修复）：**
- L67：两个"任务"格包含完整实现，`raise NotImplementedError` 完全缺失，学生面对的是演示脚本而非练习
- L71：beam_search 无任何验证断言，"分数"的定义也未给出，学生无法自我检验
- L68：推理路线给出了精确 Python 表达式（答案泄漏），认知负荷降为复制粘贴

### 5.3 边界条件覆盖

审计发现边界条件覆盖普遍不足：
- L10：零向量余弦相似度产生静默 `nan`，但测试未覆盖
- L66：`compute_wer` 对空 reference 的行为（返回 `float('inf')` vs 抛出异常）在 docstring 和正文中均未说明，仅在 solutions 文件中体现
- L69：没有测试 `labels=[]`（空序列）或 `T < 2L+1`（不可对齐输入）

**最佳模块：Audio DSP（L32–L53）**，其练习设计最接近"工程验收标准"：每个实现函数都有精确的数值验证，误差阈值（1e-10）与"研究工程师会接受的精度"对齐。

**最差模块：ASR（L66–L75）**，系统性练习缺陷（L67 答案泄漏、L68 提示泄漏、L69 整体不可运行）使该模块无法作为有效的主动学习工具。

## 七、模板化风险评估

### 7.1 模板的识别

70% 的 B- 评级并非随机分布，而是对以下固定模板的评分结果：

```
每节课标准结构（7段）：
1. 剧情（目标设定，1段 markdown）
2. 符号入口（核心公式/概念，1-2段 markdown + 1个演示格）
3. 动手观察（逐步示例，2-3个代码格）
4. 代码实验（TODO练习，1-2个格）
5. 参数实验（调参探索，1个格）
6. Aurora 连接（src/ 路径引用，1段 markdown）
7. 本课收束（总结 + 下节预告，1段 markdown）
```

此模板在 L04–L19、L23–L36、L40–L53 等课节中高度一致，即便跨越"正弦波生成"（L33）和"FFT 暴力实现"（L37）这样认知负荷差异极大的课节，也使用相同的页面布局和结构密度。

### 7.2 模板化的教学代价

**优点（不可否认）：**
- 学生能快速定位每节课的练习位置（统一体验）
- Aurora 连接的系统性保证确保了代码库与笔记本的对齐
- 可复制性高，便于快速批量生成新课节

**代价（实质性的）：**

1. **节奏单一化**：L10（点积，本科一年级水平）和 L69（CTC 前向 DP，顶级研究生水平）在页面布局上几乎无法区分。没有"大课"（深度推导密集课）与"小课"（直觉建立课）的区分，学生的认知节奏无法随课程核心算法的重要性自动调整。

2. **练习天花板**：模板中"代码实验"段固定为 1–2 个格，这对 L10 是合理的，对 L50（完整 MFCC 流水线，实际需要 6–8 步骤验证）是严重不足的。课程中最重要的综合练习被压缩到与入门练习相同的格数，深度被模板截断。

3. **参数实验的空洞化**：多课的"参数实验"段（第 5 节）退化为仅修改一个数值观察变化，缺乏真正的探究。审计发现 L21 的参数实验格甚至只有文字描述（无可执行代码格），L92 的 `n_trials = 3 # ✏️ TODO` 要求学生将 3 改成 5，被审计报告直接标注为"无任何学习价值"。

### 7.3 具体改进建议

**不应抛弃模板，而应分级使用：**

- **A 型课（深度推导课，约 15 节）**：L38、L39、L50、L54–L56、L69、L83 等算法核心课，应采用扩展模板（新增"推导验证"段，要求学生在 Markdown 格中写出核心递推式，再用代码格数值验证；练习格扩展到 4–6 步）
- **B 型课（实现课，约 45 节）**：维持当前 7 段模板，但每个 TODO 格强制要求精确数值断言（`np.allclose` 或明确的数值范围）
- **C 型课（直觉建立课，约 25 节）**：可视化主导的课节（L08、L19、L20、L26 等），允许减少 TODO 段，增加"观察记录"Markdown 格，让学生用文字描述观察到的现象
- **D 型课（收束课，约 14 节）**：L52、L65、L75、L91、L94 等模块收尾课，增加"综合验证"段，将当前模块的全部 API 组合成一个端到端测试
