### 【老陈问题5】n_frames = 1 + L // hop 的推导存在数学错误

**问题描述**：原推导写"帧数 = (L + n_fft) // hop + 1 ≈ 1 + L // hop（当 L >> n_fft 时精确成立）"。代入本课参数 L=16000, n_fft=1024, hop=256：`(16000+1024)//256+1 = 67`，而 `1+16000//256 = 63`，两者相差 4——这不是"近似"，是推导本身漏了一步，误导学生以为公式只是"在特定条件下近似成立"。

**改动位置**：`392d77a7`（《公式全景》→《常见问题澄清》→"❓ 问题4"）

**改动内容**：把原来的近似式推导替换为三步精确推导：
1. **Padding 长度**：`center=True` 时两端各填充 `n_fft//2`，填充后总长 `L' = L + n_fft`
2. **栅栏计数**：`n_frames = floor((L' - n_fft)/hop) + 1`（窗口最后一个合法起点是 `L'-n_fft`）
3. **代入相消**：`L' - n_fft = L` 精确成立（不是近似），因此 `n_frames = 1 + L//hop` 是**精确公式**，不是"L 远大于 n_fft 时的近似"

并给出正反两种算法的数值对比：
- ❌ 错误算法（漏减 n_fft）：`(L+n_fft)//hop+1 = 67`
- ✅ 正确算法：`(L'-n_fft)//hop+1 = 1+L//hop = 63`

明确指出"上一版推导遗漏了第二步减去 n_fft 这个环节"，并保留原有的"回 L44 看 center=True 详细推导"的指引。

**验证**：已运行 `7a7385b6` 演示代码，实际输出 `n_frames = 1 + 16000 // 256 = 63`，与修正后的推导完全一致，且与闭卷检查格（`c7a69e2b`/`ed3494ec`）中的已有公式没有冲突。
