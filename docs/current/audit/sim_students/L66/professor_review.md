### 老陈问题 2：Transformer encoder/decoder 为什么这样命名？

**问题**：上一轮只解释了 encoder/decoder 的功能，没解释"编码/解码"这两个字本身的含义和命名原因。

**改动内容与位置**：
- **插入新 Markdown cell（新 id `b9d6fbe4`）**，标题"补充：为什么叫「编码器」「解码器」？"
- 位置：紧跟在 cell `bf3accaa`（"1. 端到端 ASR vs 传统 HMM-GMM"，也是"编码器/解码器"这两个词首次出现的段落）之后
- 内容：
  - 类比"寄快递"：打包（把易碎品包成统一尺寸的包裹）≈ encode；拆包（还原成可用的东西）≈ decode
  - 精确对应到 ASR：encoder 把不定长的音频帧序列，转换成每帧统一维度（如 512 维）的隐状态——不管原始音频多长，每帧的"编码"维度固定；decoder 拿着这些隐状态 + 已生成文字，逐步"翻译"回具体的文字 token
  - 解释为什么不叫"前半部分/后半部分"：encode/decode 精确表达了"抽象化"与"具象化"这一对相反的操作方向，而"前/后"只表达顺序

### 老陈问题 6：为什么编辑距离操作代价统一为 1？

**问题**：转移方程里删除/插入/替换代价都设为 1，未解释这个选择的原因和局限。

**改动内容与位置**：
- **插入新 Markdown cell（新 id `453e6034`）**，标题"补充：为什么删除、插入、替换的代价都统一是 1？"
- 位置：紧跟在 cell `02dba5dd`（转移方程首次出现的"4. ✏️ 实现 compute_wer"段落）之后
- 内容：
  - 用"weather→whether"（看似相近）vs "weather→banana"（完全无关）设问，制造认知冲突
  - 回答：统一代价 1 是"标准编辑距离"（最少操作数）本身的定义，不是疏漏
  - 三点原因：避免主观争议（怎么给"相近程度"打分见仁见智）、保持数学性质干净（三角不等式，DP 正确性）、对 WER 任务本身够用（下游只关心"错了几个词"，不关心"错得离谱与否"）
  - 指出如果确实需要区分"离谱程度"，对应概念叫"加权编辑距离（weighted edit distance）"，但那是标准版之外的话题，本课/L67 只做统一代价版本

**改动内容与位置**：

- **另插入一个可运行代码 cell（新 id `6e367ef7`）**，位置在 `bf269f87` 之后
