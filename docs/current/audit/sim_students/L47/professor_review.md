### 【问题】误差对齐的 1e-6 标准从何而来（琳达问）

**学生卡点**：检查函数的标准是 `np.allclose(out, ref, atol=1e-6)`。但这个 1e-6 是怎么确定的？是浮点 float64 的精度限制吗？如果用了稍微不同的数值顺序（比如 log 后再加常数 vs 加常数后再 log），精度会超过 1e-6 吗？

**改动内容**：在第 3 部分"eps 与均值归一化"的 CMVN 演示代码 cell（`f0b109d9`）之后、第 4 部分"实现 log_mel_spectrogram"（`20cfb8dd`）之前，**新增一个完整子节 "3.1 插曲：验收标准里的 `atol=1e-6` 是怎么来的？"**（markdown cell，含新增代码 cell），内容包括：

- **float64 机器精度**：用"计算器算 1/3 只能存有限位"类比引出浮点数舍入误差的本质；给出 `np.finfo(np.float64).eps ≈ 2.22e-16` 的具体数值。
- **多步运算的误差累积**：明确列出流水线的 5 步（STFT → 取模 → 平方 → 矩阵乘法 → log），说明每一步都引入新的舍入误差，且 `log` 函数在输入接近 0（即接近 `eps=1e-8`）时因导数 `1/x` 很大而放大误差，用具体数字演示放大效应（1e-13 相对误差 → 1e-5 绝对误差）。
- **为什么选 1e-6 而不是 1e-4 或 1e-10**：既要给"计算路径不同但数学等价"的正确实现留出安全余量（真实累积误差约 1e-10~1e-6），又要能揪出真正的实现 bug（bug 造成的误差通常是 O(0.1)~O(10) 量级）。
- **直接回应琳达的追问**：区分"公式选错"（`log(x)+c` vs `log(x+c)`，数学上不等价，是 bug）和"运算顺序不同"（矩阵乘法逐帧算 vs 整体算，数学等价，纯舍入误差），说明为什么前者会被 `atol=1e-6` 抓住、后者不会。
- **与业界对比**：提到 librosa、torchaudio 等库测试 Mel 实现时也普遍用 1e-5~1e-6 量级的容差，而非逼近机器精度的 1e-12。
- **可运行演示代码**（新增 code cell）：实测对比两种误差的量级——"运算顺序不同"（`power @ fb.T` 整体算 vs 逐帧算）产生约 `2.9e-11` 的误差，"公式选错"（`log(x+eps)` vs `log(x)+eps`，取 `x` 在 `1e-10~1e-6` 量级）产生约 `4.6e0` 的误差，两者相差约 11 个数量级，直观印证 1e-6 容差的取舍逻辑。

**改动位置**：新增 markdown cell（原 id `f8d91e0a`，插入在 `f0b109d9` 之后）+ 新增 code cell（原 id `44a7f8d8`，紧随其后）。
