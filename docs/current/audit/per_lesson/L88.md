## 概述

The notebook's conceptual framing is sound — the TF-IDF formula is stated correctly, the 10-doc DSP corpus is a nice domain tie-in, and `aurora.llm.retrieve` actually exists with a matching API. However, the exercise scaffolding is seriously deficient: the two core TODO comments each describe only a small fraction of the work required, making it likely that students implement incorrect or incomplete solutions. The "verification" cell in cell 10 always prints "✅ TF-IDF 检索验证通过" regardless of whether the student's implementation is correct, because it only runs the reference — it never compares the two. And the top-to-bottom runability fails with a NameError before even reaching the cosine_retrieve TODO.

## 亮点

- IDF formula (`log((N+1)/(df+1)) + 1`) matches the reference implementation exactly — no silent discrepancy between the notebook's stated math and the ground truth code.
- Both tokenize assertion tests are correct and would pass against the reference `aurora.llm.tokenize` implementation.
- The reference implementation in `aurora.llm.retrieve` is fully implemented and importable, so the comparison cell (cell 10) does not fail with ImportError.
- The pedagogical motivation (TF-IDF vs sentence embeddings) is concise and accurate.
- `cosine_retrieve` stub pre-computes `word_idx`, `tokens`, `counts`, and `total`, reducing boilerplate so students can focus on the core math.

## 问题列表

### 🔴 Critical

**[Runability]** — Cell 9 references `matrix` and `vocab`, which are only assigned if cell 7 completes without error. Cell 7 raises `NotImplementedError`, so running the notebook top-to-bottom causes `NameError: name 'matrix' is not defined` in cell 9 — not the pedagogically useful `NotImplementedError` from inside the student's function. Students see a confusing error that has nothing to do with their implementation gap.
Fix: Guard the test block in cell 9 with `if 'matrix' in dir(): ...`, or move the test into a separate cell with an explicit `# Run only after implementing build_tfidf` comment, or initialize `matrix = None; vocab = []` as sentinel values after cell 7's stub.

**[Exercise Design — build_tfidf]** — The TODO comment in cell 7 reads "TODO: 计算 TF 矩阵 (n_docs, vocab_size)" — it says only "TF matrix". The function must compute TF, IDF, and return `TF × IDF`. A student who follows only the comment will compute and return the raw TF matrix, which is incorrect but produces the right shape. There is no assertion on values, so the shape print will appear to confirm success.
Fix: Change the comment to "TODO: (1) 计算 TF 矩阵 (n_docs, vocab_size)；(2) 计算 IDF 向量 (vocab_size,)；(3) 返回 TF × IDF 矩阵及 all_terms 词汇表列表" and add brief formula reminders pointing back to cell 1.

**[Exercise Design — cosine_retrieve]** — The TODO in cell 9 reads "TODO: 构建查询向量". The full implementation requires four steps: (1) build q_vec from TF counts, (2) compute dot products with all document rows, (3) normalize by both query and document L2 norms, (4) select and sort top_k. Three of the four steps are completely absent from the hint. A student who builds only the query vector and returns it will get a type error from the test code with no useful guidance.
Fix: Replace the single TODO with four numbered sub-TODOs, each with a formula reference or hint (e.g., "# (3) 余弦相似度: score_i = (tfidf_matrix[i] · q_vec) / (‖tfidf_matrix[i]‖ · ‖q_vec‖)").

### 🟠 Major

**[Exercise Design — hollow verification]** — Cell 10 always prints "✅ TF-IDF 检索验证通过" because it only runs the reference implementation. It never compares student output against the reference. If the student's `build_tfidf` returns raw TF (missing IDF), cell 10 still prints the success checkmark. This is a false signal that actively harms learning.
Fix: Add `assert np.allclose(matrix, ref_mat, atol=1e-5), "矩阵与参考实现不一致"` and a similar comparison for retrieval result order before printing the success message.

**[Pedagogical Quality — missing cosine similarity formula]** — The notebook states the TF-IDF formula explicitly in cell 1 but never states the cosine similarity formula anywhere before Task 3 asks students to implement it. Students with no prior IR background have no reference.
Fix: Add a formula block to the cell 8 markdown: `cos(q, d) = (q · d) / (‖q‖ · ‖d‖)`, with a one-sentence note about how zero-norm handling is done (return score 0 if query is empty).

**[Accuracy — asymmetric encoding unexplained]** — Cell 1 states "查询向量：用同样的 tokenize + TF 计算（不乘 IDF）". Documents are encoded as TF×IDF but the query is encoded as plain TF. This asymmetry is a deliberate IR design choice (sometimes called "tf · idf weighting for documents, tf-only for queries"), but it is not explained or justified. Students may implement it symmetrically (applying IDF to the query too) and get different results from the reference without understanding why.
Fix: Add one sentence: "查询向量不乘 IDF 的原因：IDF 对高频词的惩罚在文档端已经完成；对查询再次施加会过度抑制常见关键词，实验表明不乘效果更好（BM25 同样采用此设计）."

### 🟡 Minor

**[Accuracy — log base unspecified]** — The IDF formula in cell 1 writes `log(...)` without specifying the base. The reference uses `np.log` (natural log, base e). A student using log base 10 will produce different values that fail any numerical comparison.
Fix: Write `ln(...)` or `log_e(...)` and note "Python 中对应 `np.log()`" in the formula block.

**[Exercise Design — no value assertion in build_tfidf test]** — The only check after the build_tfidf stub is a shape print. If a student's IDF is inverted (e.g., `df/N` instead of `N/df`) the shape is still correct and the error is invisible.
Fix: Add `assert matrix.shape == (len(DOCS), len(vocab))` and `assert matrix.min() >= 0 and matrix.max() > 0`, and after implementing: `assert np.allclose(matrix, ref_mat, atol=1e-5)`.

**[Pedagogical Quality — learning objectives not numbered]** — Cell 0 lists four learning objectives as bullet points. Other lessons in the series number their objectives, making it easier to cross-reference in the summary table.
Fix: Number the four objectives (1)–(4).

**[Pedagogical Quality — no Aurora连接 section]** — Other lessons include a dedicated "Aurora连接" section naming the source file and explaining how the lesson's implementation maps to the production code. This lesson jumps directly from the exercise to a raw `from aurora.llm import ...` without a framing section.
Fix: Add an "Aurora连接" markdown cell before cell 10 pointing to `src/aurora/llm/retrieve.py` and noting that this module is used by the RAG pipeline in L89.
