## 概述

The core LoRA math (B @ A decomposition, zero initialization of B, alpha/rank scaling) is correct and well-explained. The exercise stub is properly structured with `raise NotImplementedError`, and the four verification checks in cell 11 cover the most important behavioral properties. However, the notebook has a phantom Aurora connection (both referenced files don't exist in the codebase), and the parameter-efficiency comparison experiment in cells 12–13 is architecturally broken: the "全参数微调" baseline uses a d→1 model (65 params) while the LoRA variants use a d→d→1 model (512–2113 params), so the "~6% savings" claim is inverted in the actual experiment.

### 🟠 Major

**[Experiment Design / Accuracy]** — The parameter-efficiency experiment (cells 12–13) compares architecturally incompatible models.  
"全参数微调" is `nn.Linear(d=64, 1)` — a d→1 model with **65 trainable parameters**.  
The LoRA models are `LoRALinear(64, 64) + nn.Linear(64, 1)` — a d→d→1 model with **577–2113 trainable parameters** depending on rank.  
For rank=16 the LoRA variant has ~33× *more* trainable parameters than the "全参数" baseline, directly contradicting the lesson's premise of "0.1% parameter savings".  
The claim "rank=16 的可训练参数仅是全参数的 ~6%" is calculated for d=512 (2×512×16 / 512² ≈ 6.25%) but the experiment runs with d=64.  
**Fix:** Replace the baseline with a model using the same d→d→1 architecture (frozen d×d layer + trainable final layer), or replace the LoRA branch with `LoRALinear` replacing only the final d→1 projection so both models have the same architectural depth. Re-derive the parameter counts for d=64 in the markdown table.

**[Exercise Verification]** — The test battery in cell 11 cannot detect an incorrect or missing `self.scale`.  
A student who writes `self.scale = alpha` instead of `alpha / rank`, or who omits the scale multiplication in `forward`, passes all four checks because none of them probe the magnitude of the LoRA contribution.  
**Fix:** Add a fifth check, for example:
```python
# 检查 5: scale factor 正确 (alpha/rank)
l_scale = LoRALinear(64, 64, rank=4, alpha=8.0)
assert abs(l_scale.scale - 2.0) < 1e-6, f"scale 期望 2.0，得到 {l_scale.scale}"
print(f"✅ scale = alpha/rank = 8.0/4 = {l_scale.scale}")
```

### 🟡 Minor

**[Runability]** — Cell 13 has no `try/except NotImplementedError` guard.  
If a student runs the notebook sequentially before completing the exercise, cell 13 crashes with an uncaught `NotImplementedError` when `LoRALinear(d, d, ...)` is constructed, with no helpful message.  
**Fix:** Wrap cell 13's construction and `run_experiment` calls in `try/except NotImplementedError: print("⬜ 请先完成 LoRALinear 练习")`.

**[Accuracy — Parameter Counts]** — Cell 8 uses `nn.Linear(d, d)` which includes a bias term, so `total = 262,656` (not 262,144). But the LoRA param formula `r * d + d * r` omits any bias analogue. The "节省" percentages are therefore computed against a slightly inflated denominator. This is a minor distortion but could confuse a student who is counting carefully.  
**Fix:** Either add `bias=False` to the `nn.Linear(d, d)` in cell 8, or note that the denominator includes bias and adjust the formula to `r * (d_in + d_out) / (d_in * d_out + d_out)`.

**[Missing Concept — Weight Merging]** — Cells 1 and 14 both claim "推理时可直接合并回原权重，推理成本为零", but this capability is never demonstrated. Students are told the key inference optimization exists without seeing how `W_merged = W₀ + (alpha/r) * B @ A` is computed or that the merged linear layer produces identical outputs to the un-merged LoRALinear.  
**Fix:** Add a brief `merge_weights()` method to the LoRALinear reference solution (or as an optional bonus exercise) and verify `torch.allclose(merged(x), lora_linear(x))`.

### 💡 建议

- **Learning objectives**: Cell 0 gives a single inline sentence for the learning objective. Other strong lessons in this course use a numbered list (e.g., "1. 理解 ΔW 的低秩假设；2. 实现 LoRALinear；3. 验证无扰动启动"). Adopt the numbered format for clarity and parallel structure.
- **Gradient check in cell 6**: The demo verifies `lora_B.grad` is non-zero even when `lora_B = 0`, which is a subtle point. Consider adding one sentence explaining *why* the gradient doesn't vanish: the chain rule passes the gradient through `B` to the accumulated activation `x @ A.T`, which is non-zero.
- **Alpha default convention**: The stub default `alpha=1.0` with `rank=8` yields `scale=0.125`, a very small multiplier. The original LoRA paper and common practice set `alpha=rank` (scale=1.0). Add a note or change the default to `alpha=None` with `if alpha is None: alpha = rank` to surface this convention explicitly.
