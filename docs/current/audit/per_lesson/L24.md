## 问题列表

### 🔴 Critical

**[Exercise Design / Runability]** — Stub `return None` causes `TypeError`, not `AssertionError`.
Cell `96ba3dcb` stubs the exercise with `return None`. The test in `95883685` immediately evaluates `abs(composite_derivative(x) - numeric)`, which raises `TypeError: unsupported operand type(s) for -: 'NoneType' and 'float'`. Students see a cryptic runtime crash rather than a clean "not yet implemented" signal.
Fix: Replace `return None` with `raise NotImplementedError("TODO: implement composite_derivative")`.

### 🟠 Major

**[Accuracy]** — `cell-4` description claims x = π/4 but the code uses x = 1.3.
The markdown cell immediately before the first demo code says "在 `x = π/4` 处计算…". The code cell `dc8b8a9f` uses `x = 1.3` (π/4 ≈ 0.785, difference ≈ 0.51). Students will be confused when the stated reference point does not match what executes.
Fix: Align the prose — either change the description to "x = 1.3" or change the code to `x = np.pi / 4`.

**[Accuracy / Pedagogical Quality]** — Cells `cell-7` and `cell-9` demonstrate simple polynomial differentiation, not chain rule, but their markdown headers frame them as chain-rule exercises.
`cell-7` computes slopes of `f(x) = x²` and `cell-9` of `f(x) = x² + 2x`. Neither involves function composition; both are straightforward power-rule cases. Headers call these "动手观察：变化率不是一句口号" and "代码实验", giving no signal that they are plain polynomial derivatives — not chain rule. This dilutes the lesson and may actively mislead students about what counts as a chain-rule application.
Fix: Either replace both cells with genuine composition examples (e.g., (x²+1)³ or e^(x²)) or add an explicit callout that these warm-up cells establish the numerical-differentiation technique before applying it to composite functions.

**[Course Continuity / Accuracy]** — 本课收束 claims numerical error is "1e-9 量级内" but the exercise assert threshold is `< 1e-3`.
The closing summary (`cell-18`) states: "数值误差在 `1e-9` 量级内". The only assertion a student sees is `assert abs(composite_derivative(x) - numeric) < 1e-3` — four orders of magnitude looser. The actual error for a correct implementation is ~2.24e-10, but the assert does not enforce this. Additionally, cell `cell-16` markdown says "误差应在 1e-9 以内" while cell `cell-15` asserts `< 1e-5`.
Fix: Change the exercise assert to `< 1e-6` to match the lesson's stated claim; update cell `cell-15` assert to `< 1e-9` to match cell `cell-16`.

**[Runability / Pedagogical Quality]** — Code cell `cell-15` appears before its explanatory markdown `cell-16`.
Cell `cell-15` (code that verifies `sin(exp(x²))` at three points) is rendered before `cell-16` (the markdown that sets up the three-layer derivation, defines intermediate variables u, v, y, and shows the full chain-rule formula). A student reading top-to-bottom sees the code first and the derivation second.
Fix: Swap the order so `cell-16` (explanation) precedes `cell-15` (code).
