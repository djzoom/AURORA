## 概述

L78 introduces spectral-flux onset detection and autocorrelation-based BPM estimation — a well-chosen topic for this point in the music section. The theory cells are accurate and concise. However, the notebook has a fundamental runability flaw: both TODO stubs immediately raise NotImplementedError and are then immediately called in the same cell, which kills execution before any student output is seen. Additionally, Task 2's scaffold is severely under-specified — beat *position* extraction is never explained yet is expected as a return value.

## 亮点

- The spectral flux formula `flux[t] = Σ_k max(0, |S[t][k]| - |S[t-1][k]|)` is stated correctly and matches the reference implementation (`np.maximum(0, np.diff(mag, axis=0)).sum(axis=1)`).
- `aurora.music.beat_track` / `aurora.music.onset_envelope` are both real, importable, and do exactly what the notebook claims — the Aurora connection in Cell 8 is valid and instructive.
- The synthetic 120-BPM test signal is well-constructed: clean beat_period arithmetic, realistic 20 ms pulse width, normalised amplitude. It should yield a strong autocorrelation peak.
- The BPM tolerance in the assertion `abs(bpm - 120) < 15` is appropriately loose for a simple autocorrelation estimator on a synthetic signal.

## 问题列表

### 🔴 Critical

**[Runability]** — Both TODO stubs raise `NotImplementedError` and are called in the *same* cell immediately after definition. Cell 5 calls `env = my_onset_envelope(signal, SR)` right after the stub; Cell 7 calls `bpm, beats = my_beat_track(signal, SR)` right after its stub. Running top-to-bottom causes an uncaught `NotImplementedError` in Cell 5, which means `env`, `times`, `bpm`, and `beats` are never defined. Cell 8 then fails with `NameError: name 'bpm' is not defined`. A fresh student cannot run or read the notebook interactively in any useful state.

Fix: Split each cell into two: one **definition cell** (the stub alone) and one **execution + test cell** wrapped in a try/except NotImplementedError that prints a friendly "implement me first" message. This is the standard Jupyter pattern for homework stubs.

### 🟠 Major

**[Exercise Design — Under-scaffolded Task 2]** — `my_beat_track` must return `(bpm, beats)` where `beats` is an array of beat times in seconds. The TODO hint only says "compute autocorrelation for each lag (dot product of env with shifted env)" — it says nothing about how to (a) identify the best lag, (b) convert lag to BPM, or (c) find actual beat positions from that lag. The beat-position step requires a non-obvious algorithm: find the argmax of env within the first period as the initial phase, then stride forward by `best_lag`. Without a hint, students will compute BPM correctly but return an empty or wrong `beats` array, and the subsequent `len(beats)` comparison will fail silently.

Fix: Add substep comments inside the function body: `# 1. ac = [np.dot(env[:-lag], env[lag:]) for lag in lags]`, `# 2. best_lag = lags[np.argmax(ac)]`, `# 3. bpm = fps * 60.0 / best_lag`, `# 4. find initial phase: pos = np.argmax(env[:best_lag]); step forward by best_lag`. Add the formula `BPM = 60 × fps / lag` to the theory markdown cell.

**[Runability / Comparison Inconsistency — n_fft mismatch]** — The student stub defaults to `n_fft=N_FFT=1024`. The reference `aurora.music.onset_envelope` defaults to `n_fft=2048`. Cell 8 calls `onset_envelope(signal, SR, hop_length=HOP)` with no n_fft argument, so it silently uses 2048. The two onset envelopes (and therefore BPM estimates) are computed over different frequency resolutions. The comparison cell prints the difference as though it measures algorithmic correctness, but it actually measures the combined effect of implementation *and* n_fft choice. On a clean implementation the BPM estimates will typically agree to within a fraction of a BPM, but the pedagogical point is muddied.

Fix: Either (a) pass `n_fft=N_FFT` explicitly to the reference calls in Cell 8, or (b) define `N_FFT=2048` to match the reference default, or (c) add a comment acknowledging the difference.

**[Exercise Design — No verification assertions for Task 1]** — Cell 5 has only `print(f'onset envelope shape: {env.shape}')` — no assert statements. Students cannot know programmatically whether their onset envelope is correct. The reference returns `(n_frames - 1,)` (one element shorter than n_frames because of `np.diff`); a student who returns `(n_frames,)` with a zero first element will see a plausible-looking plot but a wrong result, and will not be caught.

Fix: Add assertions after the plot: `assert env.ndim == 1`, `assert (env >= 0).all(), "flux must be non-negative"`, `assert env.shape[0] == stft(signal, n_fft=N_FFT, hop_length=HOP).shape[0] - 1, "should be n_frames-1"`.

### 🟡 Minor

**[Pedagogical Quality — Learning objectives not numbered]** — The spec for this course requires numbered learning objectives. The current bullet list ("- 理解…", "- 实现…", etc.) is unnumbered.

Fix: Convert to `1.`, `2.`, `3.`, `4.`.

**[Pedagogical Quality — BPM-to-lag formula absent from theory]** — The theory section explains that autocorrelation peaks at lag=τ but never states the conversion formula `BPM = 60 × (sample_rate / hop_length) / τ`. Students implementing Task 2 must derive this themselves with no guidance.

Fix: Add one line to Cell 1: `BPM = 60 × fps / τ，其中 fps = sample_rate / hop_length`.

**[Exercise Design — STFT output shape undocumented in TODO hint]** — The TODO comment for Task 1 says "调用 stft，取绝对值，计算差分，半波整流，按频率求和" but does not state the output shape of `stft()`. Checking the source shows it returns `(n_frames, n_fft//2+1)`. Without this, students doing `np.diff(mag, axis=0)` vs `np.diff(mag, axis=1)` risk a silent shape error.

Fix: Add to the TODO hint: `# stft returns (n_frames, n_fft//2+1); diff on axis=0 (time), sum on axis=1 (frequency)`.

**[Exercise Design — Expected beat count claim may mislead]** — Cell 7 prints `期望: {int(DURATION * 2)} = 8` beats. The reference implementation places the first beat at `argmax(env[:best_lag])`, not necessarily at t=0. Depending on the pulse placement and FFT centering (center=True pads by n_fft//2), the stride-forward algorithm might yield 7 or 9 beats. Presenting 8 as the hard expectation without asserting it (it's only a print, not an assert) is fine, but the number could confuse students whose correct implementation returns 7.

Fix: Change the wording to `(参考: ~{int(DURATION * 2)})` to signal it is approximate.
