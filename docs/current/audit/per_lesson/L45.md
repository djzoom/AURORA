## 概述

The notebook walks students through a clear conceptual progression from STFT complex output to magnitude, power, and dB representations, then to `imshow` visualization with correct axis labeling. The parameter experiment comparing three (win_len, hop) configurations is particularly effective. However, the exercise stub is fundamentally broken — it silently succeeds rather than raising `NotImplementedError`, so the try/except guard never fires and the student receives false confirmation that an empty implementation passes. Separately, the title promises `pcolormesh` but the entire notebook uses `imshow`, and eps values are inconsistent across three cells and the real Aurora source.

## 亮点

- The three-column parameter experiment (Cell 13) concretely demonstrates time–frequency resolution tradeoff with shared colormap scaling — this is genuinely good pedagogical design.
- Cell 5 explains `imshow`'s `origin`, `aspect`, and `extent` parameters with precise semantics before students are asked to use them.
- Cell 7 gives exact derivations for the frequency and time axes (`f[k] = k·sr/n_fft`, `t[i] = i·hop/sr`), grounding the visualization in the math rather than hand-waving.
- The conceptual explanation in Cell 3 correctly notes the approximate equivalence of `10·log10(A²+ε)` and `20·log10(A+ε)` while flagging the eps discrepancy.

## 问题列表

### 🔴 Critical

**[Exercise Design / Runability]** — The exercise stub uses Python Ellipsis literals (`S = ...`, `A = ...`, `dB = ...`) instead of `raise NotImplementedError`. The final `...` in the function body is a no-op expression statement. When Cell 11 calls `plot_spectrogram(x_test, SR)`, the function completes silently (producing an empty `plt.subplots` figure), never raises `NotImplementedError`, and the try/except prints "✅ plot_spectrogram 验证通过" for a completely unimplemented function. Students receive false positive feedback.

Fix: Replace the three `...` stubs with meaningful scaffolding and add `raise NotImplementedError("请完成 TODO 步骤 1–3")` as the first executable line of the function body (before `dur = len(x) / sr`). This ensures the try/except correctly reports "⬜ 未实现" until the student fills in the implementation.

### 🟠 Major

**[Accuracy — Title/Content Mismatch]** — The notebook title and header cell prominently say "pcolormesh 热力图", but every visualization cell (Cells 6, 11, 13) uses `plt.imshow`. These are distinct matplotlib functions with different interfaces: `pcolormesh` accepts coordinate arrays and handles non-uniform grids, while `imshow` uses `extent` for uniform grids. Teaching `imshow` while advertising `pcolormesh` could confuse students who look up documentation or encounter `pcolormesh` in spectrogram tutorials elsewhere.

Fix: Change the title and header to "imshow 热力图". If pcolormesh is pedagogically desired (it is common in audio literature), add a brief comparison cell showing both.

**[Accuracy — eps Inconsistency Across Cells and Source]** — Three different eps conventions appear without acknowledgment: Cell 3 introduces eps=1e-10 with `10·log10` (correct for Aurora source, floor = −100 dB); Cell 9's exercise hint instructs `20·log10(A + 1e-8)` (floor = −160 dB); Cell 11's reference implementation uses the same `1e-8`/factor-20 formula. The actual `aurora.audio.mel.power_to_db()` (mel.py line 96) uses `amin=1e-10` with factor 10 — matching Cell 3 but contradicting Cells 9 and 11. The stated floor "约 −160 dB" in Cell 9 is only consistent with the Cell 9/11 formula, not with Aurora.

Fix: Pick one convention consistently. The Aurora source uses `10·log10(P + 1e-10)`. The notebook should either match it throughout, or explicitly introduce `20·log10(A + 1e-8)` as a commonly-used alternative and state that the Aurora API uses the power form.

**[Exercise Design — Student Output Never Tested]** — All assertions in Cell 11 validate `dB_ref` (computed directly from the reference implementation), not the student's variables. There is no check that calls the student's `plot_spectrogram` and then inspects its output. A student whose function returns the wrong shape or dB range would never learn this from the current test cell.

Fix: After `plot_spectrogram(x_test, SR)` is called, capture and validate its internal state, or restructure `plot_spectrogram` to return the dB array so the caller can assert on it (e.g., `assert returned_dB.shape == (513, n_frames)` and `assert returned_dB.max() > -20`).

**[Runability — Unused Imports]** — Cell 2 imports `read_wav`, `magnitude_spectrogram`, and `power_to_db`. None of these are called anywhere in the notebook. The header and conclusion both claim `magnitude_spectrogram` and `power_to_db` are the Aurora connections for this lesson, yet neither function is invoked or demonstrated.

Fix: Either remove the unused imports, or add a demonstration cell at the end showing the Aurora one-liner `magnitude_spectrogram(x)` alongside the hand-built version, confirming they agree numerically.

### 🟡 Minor

**[Pedagogical Quality — dB Formula Discrepancy with Aurora API Not Explained]** — Students are taught to use `20·log10(A + eps)` in the exercise but the Aurora API (`power_to_db`) uses `10·log10(P + eps)`. ### 💡 建议

- Consider replacing the `plot_spectrogram` return type: return the `dB` array from the function so both visual and numerical outputs can be tested by the caller. This is a common pattern in audio libraries and makes the function more useful beyond the exercise.
- The `extent` upper-right boundary (`n_frames*hop/sr`) is computed as a simple product, but when `center=True` the last frame center extends `n_fft//2` samples beyond the signal. A brief note about this edge would complete the axis accuracy explanation begun in Cell 7.
