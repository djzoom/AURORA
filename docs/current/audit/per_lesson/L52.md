## 概述

L52 作为 Audio Core 的收口课，结构清晰，数学公式准确，test count（38）经手动核对为真实值。但它在两个地方实质性地歪曲了测试套件的覆盖范围：MFCC 的测试只检查形状，并不存在"atol=1e-8"的数值比对；此外 Cell 6 的 `cwd` 路径计算依赖于 Jupyter 从笔记本自身目录启动，在常见的"从仓库根目录启动 JupyterLab"的场景下会把 pytest 指向错误目录而失败。Cell 8 还有一处潜在 NameError（`sine` 未在该 cell 中导入）。

## 亮点

- **数学公式全部正确**：Hann 周期窗公式、RFFT 频点数推导 `N//2+1`、分帧帧数公式 `1 + (T-N)//H`、DCT-II 正交归一化系数（`sqrt(1/N)` 首项、`sqrt(2/N)` 其余项）均无误。
- **Cell 8 五项交叉检验设计扎实**：DCT-II 手动公式、mel 滤波器组形状/非负性、Hann periodic/symmetric 等价性、STFT 频率维、hz↔mel 往返误差——每项都独立于实现给出参考，符合"从公式推断、不靠记忆"的课程设计原则。
- **测试数量属实**：标题声称"38 个单元测试"，经 `pytest --collect-only` 验证确为 38 项（test_features: 13，test_transforms: 25）。
- **librosa 差异分析（Cell 9-10）**：对 center padding、norm 参数、power 参数三类差异的解释准确，是面试中回答"为什么输出不一样"的模板答案。

## 问题列表

### 🔴 Critical

无。

### 🟠 Major

**[Accuracy / 测试覆盖描述]** — Cell 5 表格声称 `test_features.py` 对 MFCC 进行了 `atol=1e-8` 的数值比对。实际上 `test_mfcc_shape()` 只有一行断言：`assert C.shape[1] == 13`，根本不检查数值。这意味着 MFCC 的值正确性在测试套件中完全未被验证，而该课大篇幅渲染的"数值精度是 Month 1 通关标准"的说法对 MFCC 而言是虚假宣传。
**Fix**：在 `tests/audio/test_features.py` 中为 `test_mfcc_shape` 补充数值参考测试（可用 DCT-II + mel_spectrogram 手动拼出参考值），并将 Cell 5 中的 `atol=1e-8 (MFCC)` 更正为"仅形状断言，数值未测"或删除。

**[Runability / cwd 路径]** — Cell 6 的 `cwd=str(Path.cwd().parent.parent)` 注释写"aurora repo root"，但逻辑假设 `Path.cwd()` 是笔记本所在目录 `notebooks/5_audio_dsp/`。在 JupyterLab / 经典 Jupyter 从仓库根目录 `/Users/z/AURORA/` 启动时，`Path.cwd()` 就是仓库根，`parent.parent` 会变成 `/Users/`，pytest 找不到 `tests/audio/`，Cell 6 以 `AssertionError` 终止。（VS Code Jupyter 默认把 cwd 切到笔记本目录，该场景下正确。）
**Fix**：改为动态探测仓库根，如：
```python
import subprocess as _sp
repo_root = _sp.check_output(["git", "rev-parse", "--show-toplevel"], text=True).strip()
```
或固定写 `Path(__file__).parent.parent.parent`（notebooks 目录再上两级），再或直接用 `Path.cwd()` 并让文档明确说明须从仓库根启动 Jupyter。

**[Runability / NameError]** — Cell 8 对 `sine` 的调用（`x_1s = sine(440.0, 1.0, 16000)`）依赖 Cell 4 已经执行过（Cell 4 `from aurora.audio import sine, ...`）。Cell 8 自身的 import 块中没有 `sine`，重启内核后直接运行 Cell 8 会报 `NameError: name 'sine' is not defined`。
**Fix**：在 Cell 8 的 import 列表中加入 `sine`。

### 🟡 Minor

**[Accuracy / atol 描述]** — Cell 1 写"atol=1e-8 级别的数值比对"作为精度总结，但实际上 FFT/DFT 测试用 `atol=1e-9`，窗函数和 DCT-II 测试用 `atol=1e-12`，均比 1e-8 严格一个量级。该措辞会让学生低估精度要求。
**Fix**：改为"最严可达 `atol=1e-12`（窗函数、DCT-II），FFT 为 `atol=1e-9`"。

**Fix**：删除未使用的导入，或补充对应的验证代码（如对 `hamming` 做对应 symmetric 检验，对 `frame_signal` 验证帧数公式）。

**[Pedagogical / 无编号学习目标]** — Cell 0 的"目标"是一句话描述，缺乏结构化的编号学习目标（学完本课，学生能做到 X / Y / Z）。其他课（如 L32 起）普遍有 3-5 条具体的能力目标。
**Fix**：添加 3 条具体目标，如：(1) 能在 pytest 输出中读懂每条测试与对应公式的关联；(2) 能解释 librosa MFCC 与 aurora MFCC 数值差异的三个来源；(3) 能对着空白文件重写 `dct_ii` 并解释每行数学含义。

**[Accuracy / Cell 5 表格 atol 行]** — 表格把窗函数和 MFCC 的 atol 写在同一列，格式为"atol=1e-12 窗函数、atol=1e-8 MFCC"，实际对 MFCC 没有 atol 测试（只有形状检查），对 DCT-II 有 atol=1e-12 但不等同于 MFCC 端到端。此行信息有两处不准确。（与第一个 Major 问题相关，但作为表格错误单独列出。）
