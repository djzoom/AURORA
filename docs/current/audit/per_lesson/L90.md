## 概述

L90 targets a well-chosen deliverable — a multi-turn podcast QA engine with source attribution — and the pedagogical scaffold for the core `podcast_qa()` exercise is clear and stepwise. However the notebook is undermined by a critical API mismatch that guarantees the real `aurora.llm` code is never exercised (the import always fails and the stubs silently take over), a broken Section 5 experiment that makes RAG appear non-functional, and a wrong lesson cross-reference carried in from an earlier edit. The lesson also silently abandons the TF-IDF paradigm established in L89 without explanation.

---

## 亮点

- **`podcast_qa()` scaffold** (Cell 10) is well-designed: five numbered TODO steps each map to a named helper, giving students a clear mental model of the data flow.
- **Prompt format** (Section 1) is concisely motivated with the ordering rationale (CONTEXT before HISTORY).
- **`truncate_history`** implementation and explanation of the sliding window tradeoff is accurate and concise.
- **Stub fallback pattern** (`try: from aurora.llm … except ImportError`) is the right defensive idiom for progressive development — the intent is correct even though the specific names are wrong.

## 问题列表

### 🔴 Critical

**[Course Continuity / Runability] aurora.llm API mismatch — notebook always runs in stub mode**

Cell 2 attempts `from aurora.llm import build_rag_index, retrieve`. These names do not exist in `aurora/llm/__init__.py` (confirmed by running the import). The real module exports `build_tfidf` and `cosine_retrieve`. The ImportError is caught silently, so the stubs run every time — students can never exercise the real retrieval code, and the "✅ aurora.llm 导入成功" success path is unreachable.

Fix: Either (a) update the import to `from aurora.llm import build_tfidf, cosine_retrieve` and adapt the notebook to the TF-IDF API, or (b) add `build_rag_index` / `retrieve` as aliases/wrappers in `aurora/llm/__init__.py` that wrap the dense-embedding path. Either way the docstring comment "如果 aurora.llm 尚未就绪" is misleading — aurora.llm *is* ready, just with different names.

L89's "本课收束" says: "下一步 L90：Tool-calling Agent——让模型自己决定什么时候调用哪个工具." L90 is actually Conversational RAG. The advertised topic never appears.

### 🟠 Major

**[Pedagogical Quality / Course Continuity] Silent paradigm switch: TF-IDF → dense embeddings, unexplained**

L89 established TF-IDF / `cosine_retrieve` as the retrieval method. L90 silently switches to dense embedding retrieval via an opaque `embed_fn` parameter, using a hash-based stub that produces random vectors. No markdown cell explains why or when one prefers dense over sparse retrieval. Students following sequentially will be confused about whether the two lessons describe the same or different systems.

Fix: Add a transition paragraph in Section 1 or a new "检索方式对比" cell explaining TF-IDF vs. embedding-based retrieval and why L90 targets the latter.

---

**[Runability / Pedagogical Quality] Section 5 experiment produces all ❌ hits, making RAG appear broken**

`_StubLLM.generate()` returns `"John played lead guitar in this episode."` for every query. The five test questions expect Bob, Alice, Carol, Eve, and Hank — none of which match "John". Every hit check prints ❌. The closing observation ("有 RAG 时答案直接引用 chunk 文本") is then contradicted by the output students actually see.

Fix: Make `_StubLLM.generate()` pattern-match the retrieved chunk content and return a context-aware answer, or replace the stub with a template that echoes the first retrieved chunk's text so the hit check can pass for at least some questions.

---

**[Accuracy] `build_rag_index` stub docstring claims tuple return; implementation returns ndarray only**

Cell 2: docstring says "返回 (embeddings_matrix, chunks) 作为索引" but `return vecs` returns only the ndarray. Students reading the docstring and then using the return value as a tuple will get a shape-indexing error.

Fix: Either change the return to `return vecs, chunks` (and update `retrieve` to unpack it), or fix the docstring to say "返回嵌入矩阵 (n_chunks, dim)".

---

**[Pedagogical Quality] Missing 学习目标 section**

The notebook jumps directly to "核心直觉" without a numbered 学习目标 block. All other late-course notebooks in this repo include one; its absence breaks the structural contract students expect.

Fix: Add a 学习目标 cell after the title: (1) 实现多轮对话记忆（滑动窗口）; (2) 构建带来源归因的 RAG prompt; (3) 完成 `podcast_qa()` 端到端流水线。

### 🟡 Minor

**[Accuracy] `_stub_embed` hashes chunk dicts and query strings with different inputs, making retrieval semantically random**

When building the index, `embed_fn(chunk_dict)` hashes `chunk_dict["text"]`. When querying, `embed_fn(query_str)` hashes the raw query. Because the hash seeds differ, cosine similarities are essentially random. The Section 5 experiment therefore cannot demonstrate any real retrieval quality, even conceptually.

Fix: In the stub only, when passed a dict, use the same path as a string (hash the text), and document clearly that `embed_fn` always receives a plain string — the dict-handling branch exists only for a legacy call site in the stub `build_rag_index`.

---

**[Exercise Design] TODO steps use `...` (Ellipsis) instead of `raise NotImplementedError`**

If a student implements steps 1–3 but not step 4, `answer = ...` assigns Ellipsis, and `return answer, sources` returns `(Ellipsis, ...)`. The test cell handles `answer is ...` correctly, but if `prompt = ...` is passed to `llm.generate(...)`, Python will raise a `TypeError` (argument to `str.split()`) rather than a clear `NotImplementedError`, which is harder to diagnose.

Fix: Replace `= ...` stubs with `raise NotImplementedError("TODO 步骤 N")` inside a guarded block, or at minimum add a comment explaining the Ellipsis sentinel pattern.

---

**[Accuracy] `np.random.seed(42)` uses deprecated legacy random API (Cell 13)**

The same notebook already uses `np.random.default_rng()` in `_stub_embed`. The legacy `np.random.seed` call in the experiment cell is inconsistent.

Fix: Replace with `rng = np.random.default_rng(42)` and use `rng` for any random operations in that cell.
