## 概述

L92 的教学主线（延迟分解、错误处理三元组、asyncio 并发、流式输出、可观测性）思路清晰，`StageResult` 设计模式也实用。然而本课最根本的问题是：它声称"串联 Aurora 所有月份成果"并"形成 Aurora v1 系统闭环"，但实际引用的三个模块（`aurora.speech.whisper`、`aurora.rag.retriever`、`aurora.llm.agent`）在代码库中均不存在，全部是占位 stub 或根本不存在的子模块。整门课的集成叙事建立在幻影模块之上。

## 亮点

- **StageResult 三元组设计**（Section 2）：`(value, ok, error)` 模式清晰表达了生产级流水线的错误传播哲学，配合 `check_audio` 的 RMS 静音检测是本课最扎实的内容。
- **asyncio 并发结构**（Section 4）：先串行后并行的对照设计教学意图良好，`asyncio.to_thread` 包装同步 HF 调用的模板是实用的工程知识。
- **可观测性层级表**（Section 5）：日志→指标→告警三层框架简洁，`timed_stage` 的结构化 JSON 日志演示接地气。
- **HuggingFace Whisper 集成模板**：作为注释出现的真实集成代码（带 `generate_kwargs={"language": "zh"}` 细节），给学生一条可执行的落地路径。

## 问题列表

### 🔴 Critical

**[Accuracy / Aurora连接]** — `aurora.speech.whisper`、`aurora.rag.retriever`、`aurora.llm.agent` 均不存在。检查：`aurora.speech` 的 `__init__.py` 只有 `"""Aurora Speech Core — planned."""`；`aurora.rag` 同样是纯 stub；`aurora.llm` 只有 `kvcache.py`、`retrieve.py`、`sample.py`，无 `agent` 模块。首格 "Aurora 连接" 和 "本课收束" 均断言"完整串联了四个模块，形成 Aurora v1 系统闭环"——这是事实错误。整堂课的所有 ASR/RAG/LLM 操作全为 mock，从不调用任何 Aurora 模块。  
**Fix**: 将 Aurora 连接改为实际存在的模块（`aurora.llm.retrieve.build_tfidf / cosine_retrieve`、`aurora.llm.sample.*`）；或明确标注"以下模块计划中，本课用 mock 演示集成模式"，并从"本课收束"中删除"完整串联…形成系统闭环"的陈述。

### 🟠 Major

1. **[Accuracy / 并行加速计算错误]** — Section 4 ASCII 图声称并行后总延迟从 2550ms 降至 2500ms（节省 50ms）。但 `run_pipeline_parallel` 在 `asyncio.gather` 之后额外执行了 `async_rag(transcript, delay=0.03)`（30ms），实际总延迟为 max(500,50) + 30 + 2000 = **2530ms**，节省仅 **20ms**，不是 50ms。  
**Fix**: 修正 ASCII 图为 2530ms（-20ms），或去掉 `rag_refine` 步骤（让第一次并行 RAG 直接用于 LLM），使代码和图示一致。

2. **[Runability / 丢弃的 RAG 结果]** — `run_pipeline_parallel` 中第一次 `asyncio.gather` 返回的 `context` 变量在后续代码中从未使用（LLM 只用 `context_refined`）。这使得"预热"RAG 调用的 50ms 完全是额外开销，不是优化。逻辑上也矛盾：预热索引在真实系统中可以加速后续查询，但在 mock 中完全无效。  
**Fix**: 要么注释清楚"context 是索引预热，真实系统中会加速 rag_refine"，要么把第一次 RAG 结果直接传给 LLM（去掉 rag_refine 步骤），保持逻辑一致。

3. **[Runability / 无用导入]** — 第 2 个 cell 底部执行 `from aurora.audio.io import read_wav`，但 `read_wav` 在整个 notebook 中从未使用。该导入紧接在"如果模块缺失本 cell 会失败"的警告之后，语义混乱。  
**Fix**: 删除该导入，或在某个 cell 中实际使用 `read_wav` 加载示例 WAV 文件，体现真实录音输入。

4. **[Exercise Design / 练习 Section 6 空壳]** — "识别你的流水线瓶颈"练习要求填入实测延迟，但 notebook 中从未建立可测量的真实流水线（所有延迟都是 `time.sleep` 常数），学生根本无法"实测"。练习没有验证测试，没有 `raise NotImplementedError` stub，没有说明如何获得测量值。  
**Fix**: 将练习改为对 `simulate_pipeline()` 运行计时并用脚本自动填入结果，或明确告知学生如何使用 Section 5 的 `timed_stage` 工具量化各阶段；补充 `assert` 验证（如检查 bottleneck 是否为 `llm_generate`）。

### 🟡 Minor

1. **[Pedagogical Quality / tokens ≠ characters]** — `mock_llm_stream` 注释写"逐字符 yield 模拟 token"，但正文 Section 3 说"首 token 往往在 200ms 内到达"。字符不等于 token（GPT-4 平均约 3-4 字节/token，中文约 1-2 字/token）。用字符模拟 token 会让学生对 tokenizer 产生误解。  
**Fix**: 将注释改为"逐字符 yield 近似模拟 token 流（实际 token 可含多字符）"。

2. **[Exercise Design / 无意义 TODO]** — `n_trials = 3  # ✏️ TODO`，要求学生把 3 改成 5，无任何学习价值。  
**Fix**: 删除 TODO 标注，直接设为 5；或换成有意义的练习（如让学生观察 n_trials=10 与 3 时标准差的变化，写出结论）。

3. **[Runability / 晦涩的 lambda 模式]** — Section 5 的 `timed_stage("audio_load", lambda: time.sleep(0.01) or b"pcm")` 使用 `None or b"pcm"` 的隐式真值转换，对初学者不透明，且无注释说明。  
**Fix**: 改为显式函数或 `lambda: (time.sleep(0.01), b"pcm")[1]`，并加注释。

4. **[Pedagogical Quality / 缺少编号学习目标]** — 首格只有一句"目标"描述，无编号的学习目标列表，与课程其他章节风格不一致。  
**Fix**: 补充 3-4 条编号学习目标，如"(1) 分解端到端延迟为四阶段并量化；(2) 用 asyncio.gather 实现 ASR+RAG 并发…"。

5. **[Course Continuity / 标题承诺未兑现]** — 标题含"麦克风"，但 notebook 中没有任何麦克风录音代码（`sounddevice`、`pyaudio` 或 `aurora.audio.io` 的录音功能）。HF Whisper 集成也只出现在字符串变量中而非可运行 cell。  
**Fix**: 标题改为"端到端流水线模拟——延迟分解、asyncio 并发与可观测性"，或补充一个真实 `sounddevice` 录音 cell（即使在运行时跳过也行，标注"需要麦克风"）。
