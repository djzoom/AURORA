However, it fundamentally fails as a course exercise: both "task" cells (✏️ 任务 1 and ✏️ 任务 2) deliver fully implemented solutions rather than student stubs — there is nothing for the learner to actually write. The notebook reads as a demo script, not an exercise sheet.

## 问题列表

### 🔴 Critical

**[Exercise Design]** — Both "✏️ 任务 1" and "✏️ 任务 2" are fully pre-implemented. Cell 4 contains the complete `edit_distance()` solution; cell 7 contains the complete `wer()` solution. There are no `TODO` comments, no `raise NotImplementedError`, and no blank scaffolds. A student has nothing to implement. The ✏️ markers and "任务" labels are misleading — students may not realize they are supposed to write code.
- Fix: Replace each function body (except the docstring and type signature) with a `raise NotImplementedError("你的实现")` stub. Move the full solution to a collapsed "参考答案" cell tagged `# SOLUTION` so graders can reveal it. Keep the verification assertions as-is — they become the student's acceptance tests.

### 🟠 Major

**[Code Quality / Pedagogy]** — `wer()` duplicates the entire 12-line DP loop verbatim from `edit_distance()` instead of calling it. Because `edit_distance` already accepts any sequence (it indexes with `[i-1]`), calling it on word lists works directly. This duplication is never flagged or explained to students, inadvertently teaching bad practice.
- Fix: Implement `wer` as `return edit_distance_words(ref_words, hyp_words) / len(ref_words)`, or rewrite `edit_distance` to accept `Sequence` and reuse it. Add a comment explaining why word-level WER delegates to the same algorithm.

**[Course Structure]** — The lesson has no "Aurora连接" section. Every other lesson in the course connects the notebook content to a concrete file in `src/aurora/`. ASR modules presumably have `src/aurora/asr/` with WER utilities or CTC code, and this lesson should point students there.
- Fix: Add an "Aurora 连接" section referencing the relevant module (e.g., `src/aurora/asr/metrics.py` or equivalent) and showing how the lesson's `wer()` maps to the production implementation.

**[Pedagogical Quality]** — Learning objectives are listed as bullet points, not numbered items. The course standard (and the audit rubric) requires numbered, specific objectives.
- Fix: Renumber the four bullet points as `1.`, `2.`, `3.`, `4.` in the markdown cell.

### 🟡 Minor

**[Runability]** — `import numpy as np` (cell 2) is never used. NumPy is not needed for pure-Python DP.
- Fix: Remove the import, or replace it with a comment explaining it would be needed for a vectorized variant (if that is introduced later).

**[Exercise Design]** — The `alignment()` function has no assertion-based verification, only visual print output. If a student modifies the backtrace logic incorrectly, no automated check catches it.
- Fix: Add at least one assertion, e.g., check that the number of S/I/D operations equals the edit distance, or that the M count equals the number of matched words.

**[Pedagogical Quality]** — Computational complexity is mentioned only in the summary table (`O(m×n)`) without explanation. Space complexity is O(m×n) as implemented, but the well-known O(n) two-row optimization is not mentioned. At lesson 67 in a research-oriented course, omitting this is a missed teachable moment.
- Fix: Add a brief paragraph (3–5 lines) noting O(n) space optimization and explaining when it is appropriate (when you only need the distance, not the backtrace path).

**[Exercise Design]** — The WER test suite does not explicitly test the `float('inf')` branch that the code handles (`len(ref_words) == 0` and `len(hyp_words) > 0`).
- Fix: Add `assert wer('', 'hello') == float('inf')` and `assert wer('', '') == 0.0` to cell 8.

### 💡 建议

- Consider adding a small DP-table visualization (print the 2D array for a short example like `"abc"` vs `"ac"`) so students see concretely how the table fills in. This bridges the recurrence formula and the code.
- The `alignment()` function hardcodes word-level splitting; noting this (and that a character-level variant would replace `.split()` with `list()`) would reinforce the generality of the algorithm.
- A timing cell comparing naive recursive edit distance (exponential) vs. the DP version would motivate the DP approach more strongly, appropriate for a lesson explicitly subtitled "动态规划."
