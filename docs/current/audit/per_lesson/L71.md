## 概述

## 问题列表

### 🔴 Critical

**[Exercise Design / Runability]** — `result` NameError 风险。Cell 5 在 `greedy_decode` 未实现时抛出 `NotImplementedError`，`result` 变量从未被绑定。Cell 7 随后执行 `print(f'贪婪结果: {result}')` 将触发 `NameError`，报错信息会掩盖学生真正要看的 beam search 输出，造成调试混乱。Fix：在 cell 5 调用前用 `try/except NotImplementedError` 捕获并给 `result` 赋占位值，或将两个打印语句分别置于各自任务 cell 内。

**[Exercise Design]** — beam_search 任务零断言、规格不完整。`beam_search` 函数只有一行 docstring "返回分数最高的序列"，但从未定义"分数"是什么（log 概率之和？归一化？），学生没有任何验证手段，无法知道自己的实现是否正确。这是整个课程所有 TODO 任务中约束最弱的一个。Fix：在 docstring 中明确"分数 = 该路径所有 log P(token_i) 之和"，并添加至少两个断言：（1）返回值以 prompt 为前缀；（2）beam_result 的 log-prob 和 ≥ greedy_result 的 log-prob 和（beam search 保证不劣于贪婪）。

### 🟠 Major

**[Accuracy]** — 贪婪解码复杂度前后矛盾。Cell 1 正文写"快，O(T)"，Cell 8 汇总表写"O(T×V)"。表中更准确——每步需扫描全部 V 个 logit 取 argmax。Fix：将 Cell 1 中的 "O(T)" 修正为 "O(T×V)"，或补注"V 为词汇表大小，此处 V=10 可视为常数"。

**[Pedagogical Quality]** — 缺失 Aurora连接 section。L70 和 L72 均有明确的 "🔗 Aurora 连接" 说明，L71 完全没有。`aurora.llm` 模块（参见 L86 的 `aurora.llm.top_k_sample`）或 `aurora.audio` 的解码接口均与本课相关。Fix：补充一段说明本课实现的 `greedy_decode` 和 `beam_search` 与 `aurora.llm` 中哪个接口对应，或说明 Aurora 的 ASR 推理流水线在哪个模块调用解码器。

**[Exercise Design]** — greedy_decode 断言不足。唯一断言 `result[0:3] == prompt` 只验证前缀保留，不验证：（1）每步确实选择 argmax（而非随机采样）；（2）遇到 EOT 后停止；（3）达到 MAX_STEPS 时正常返回。Fix：增加 `assert result[-1] == EOT or len(result) - len(prompt) == max_steps` 验证终止条件，以及对固定种子下已知 argmax 序列的确定性检验。

**[Exercise Design]** — 两个 TODO 均无推理提示（hint）。学生面对空白函数，没有算法步骤骨架。对 beam search 尤甚——这是本课程中首次出现 beam search 实现，难度足以让初学者在空白页上卡住。Fix：在 greedy_decode 的 TODO 注释中补 3 步提示（取 logits → softmax → argmax → append → 检查 EOT）；在 beam_search 中补 4 步提示（初始化 beams 为 [(0.0, prompt)] → 扩展 W×V → 取 top-W → 遇 EOT 停止）。

### 🟡 Minor

**[Pedagogical Quality]** — 课程内容过于单薄。8 个 cell 中 2 个是任务标题 markdown、2 个是 stub，实际教学内容仅 4 个 cell。对 beam search 的直觉说明（为何比贪婪更好、何时实际更差）完全缺失。Fix：补充一个对比可视化：对相同 fake_lm，绘制贪婪路径与 beam search 路径的 log-prob 对比。
