# L63 — 音频分类模型 — CNN + Mel 特征，在 Speech Commands 上定义网络

**总体评级**: B- | **问题数**: 1 critical / 4 major / 5 minor

## 概述

L63 introduces the `KeywordCNN` architecture cleanly: the full-frequency conv1 trick, global average pooling motivation, and variable-T robustness are all explained with correct code. However, the lesson has no numbered learning objectives, a stale next-lesson reference using the old "M2-K3" module/key naming instead of an L-number, a factual inconsistency between the claimed T=101 and the T=81 used everywhere in code, and a title that promises "Speech Commands" content the lesson never delivers. The TODO stub also uses bare `pass` instead of `raise NotImplementedError`, producing a confusing `AttributeError` when students first run the check cell.

## 亮点

- **Full-frequency conv1 rationale** is correctly explained: `kernel_size=(n_mels, 5)` collapses the frequency axis in a single pass, making the operation equivalent to a 1D temporal conv over all mel bands simultaneously. The shape trace `(B,1,n_mels,T) → (B,32,1,T)` is accurate.
- **GAP motivation** is the best paragraph in the notebook: the argument that parameter count is independent of T, and the variable-T demos in Cell 8 and Cell 11 directly validate this property.
- **Architecture scaffolding** is thorough: each sub-component is shown working in isolation before the student is asked to assemble the whole.
- **Receptive field experiment** (Cell 13) is a genuinely useful ablation exercise and the RF formula `k + 3 - 1` is numerically correct for stride-1 convolutions with this specific two-layer stack.

## 问题列表

### 🔴 Critical

**[Runability]** — `KeywordCNN.forward` returns `None` (bare `pass`), so Cell 11's check cell throws `AttributeError: 'NoneType' object has no attribute 'shape'` rather than a meaningful `NotImplementedError`. Students will be confused because the error message points at the assertion line rather than at their unimplemented method. `__init__` also uses bare `pass`, which means the `super().__init__()` call never runs, making PyTorch's module machinery silently broken until the student begins implementing.
Fix: Replace both stubs with `raise NotImplementedError("TODO: implement KeywordCNN")`.

### 🟠 Major

**[Course Continuity]** — The closing "本课收束" cell references `下一节（M2-K3）` — a stale module/key identifier from an earlier naming scheme — instead of the lesson's L-number. Every other lesson in the course uses `L##` references. If the next lesson is L64, this should say "L64".
Fix: Replace `（M2-K3）` with the correct `L64`(or whichever L-number follows).

**[Accuracy — Numerical Inconsistency]** — Section 1 states the "typical value" is `T=101` for 1 s @ 16 kHz + hop=160, but every code cell (4, 6, 8, 11, 13) uses `T=81`. The two values are irreconcilable without knowing the exact window/padding settings of `aurora.audio.mel.mel_spectrogram()`. The markdown number was probably copied from a different parameterization and never updated.
Fix: Either derive T=81 explicitly (e.g. `T = ceil(n_samples / hop)`) or change the code examples to T=101 and show why.

**[Exercise Design — Test Completeness]** — Cell 11 only tests `n_classes=10`. A student who hardcodes `Linear(64, 10)` instead of `Linear(64, n_classes)` will pass all checks. There is no test with a different class count.
Fix: Add `out3 = KeywordCNN(40, 35)(torch.randn(2, 1, 40, 81)); assert out3.shape == (2, 35)` — the Speech Commands full-set size makes this test pedagogically motivated.

**[Pedagogical Quality — Missing Learning Objectives]** — The lesson has no numbered 学习目标 section. The single one-line `目标:` in the header cell is insufficient; it only lists what the model does, not what the student will be able to do or understand after the lesson.
Fix: Add a markdown cell early in the notebook with 3–4 numbered objectives (e.g., "解释为何 kernel=(n_mels,k) 能在单次卷积中压缩频率维度", "实现并验证 KeywordCNN 的 forward 方法", "分析感受野与 kernel 宽度的定量关系").

### 🟡 Minor

**[Course Continuity — Title/Content Mismatch]** — The lesson title says "在 Speech Commands 上定义网络" but the dataset is never introduced, referenced, or loaded. The lesson is purely architecture definition. The title overpromises.
Fix: Change to "定义 KeywordCNN 卷积分类网络" or move dataset introduction into this lesson.

**[Accuracy — Receptive Field Formula Scope]** — Cell 5 states `RF = kernel_width + (layers-1)×(kernel_width-1)` as a general formula, but this only holds when all layers use the same kernel width. The actual architecture uses mixed kernels (k and 3). The formula is correct for the single-layer conv1 case described, but the parenthetical "(layers-1)" term implies generality that does not hold.
Fix: State the formula only for the single-layer case, then give the two-layer formula separately as `RF = k1 + (k2 - 1)`.

**[Pedagogical Quality — Unexplained Operators]** — `ReLU` and `MaxPool2d` are used in the architecture spec (Cell 9) and experiment (Cell 13) without any explanation. For a student encountering PyTorch for the first time, at minimum a one-sentence description of each is needed.
Fix: Add a brief aside: "ReLU 引入非线性，MaxPool2d 沿时间维下采样（步长2），使后续层的感受野翻倍".

**[Exercise Design — make_cnn_with_k Independence]** — The kernel experiment in Cell 13 builds its own `nn.Sequential` rather than using the student's `KeywordCNN`. The student's implementation is never exercised in the experiment, so a broken `KeywordCNN` would not be caught here.
Fix: Refactor `make_cnn_with_k` to accept `kernel` as a parameter and instantiate `KeywordCNN` (or extract the conv stack from it), so the experiment validates the student's work.

**[Course Continuity — Missing Back-Reference]** — The Aurora Connection mentions `aurora.audio.mel.mel_spectrogram()` but does not cross-reference the lesson(s) where it was introduced (presumably L32–L40 in the audio DSP module). Students who need a refresher have no pointer.
Fix: Add "(参见 L3X)" with the correct lesson number.

**[Accuracy — MaxPool RF Effect in Experiment]** — Cell 12's experiment description says the two-layer RF is `k + 3 - 1`, but the `make_cnn_with_k` network includes `MaxPool2d((1,2))` after conv2. MaxPool with stride 2 doubles the effective stride, which means any future layer's receptive field at the input is doubled. The experiment glosses over this by computing RF only for the conv layers, which is correct for this isolated stack but could mislead students who apply the formula to deeper architectures.
Fix: Add a note: "MaxPool2d 不增加此处的感受野，但若在池化后再接卷积，后续层每一步对应输入的 2 帧，相当于感受野按步长倍增".

### 💡 建议

- Add a `model.count_parameters()` helper and print the parameter count for the reference architecture (~15K params). This makes the "lightweight" claim concrete and is a natural lead-in to ablation.
- The experiment would be more impactful with a real 1-second waveform piped through `mel_spectrogram()` → `KeywordCNN`, even if the weights are random, to show the full pipeline end-to-end. This also validates that `aurora.audio.mel` output shape is compatible with the model input convention.
- Consider adding `BatchNorm2d` after each conv as a comment-out option, with a brief note that it stabilizes training — giving students a path to improve the baseline architecture in the next lesson.

## 优先修复（前3项）

1. **Replace `pass` stubs with `raise NotImplementedError`** in both `__init__` and `forward` — this is a correctness/UX issue that will immediately confuse every student who runs the notebook before finishing the exercise.
2. **Add `n_classes=35` shape test** to Cell 11 — the current tests allow a hardcoded-10 implementation to pass silently, which means students may carry a broken implementation into the next lesson where training actually happens.
3. **Fix the T=101 / T=81 inconsistency** in Section 1 — the claimed "典型值" should match the code, and ideally be derived from `aurora.audio.mel.mel_spectrogram()` parameters documented elsewhere in the course.
