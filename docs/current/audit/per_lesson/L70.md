## 概述

L70 has solid pedagogical bones — the Aurora-vs-Whisper parameter comparison table, the cross-attention shape demo, and the scaffolded `whisper_preprocess` exercise are all well-conceived. However, the lesson contains a factual self-contradiction about Whisper's encoder downsampling factor that will directly mislead students studying the architecture, a garbled lesson-closing paragraph with a dangling backtick that breaks the terminal markdown, and a missing "多任务头" section despite its prominent placement in the title and lesson header. The TODO stub's use of `pass` instead of `raise NotImplementedError` causes a confusing `AttributeError` rather than a clear signal that implementation is pending.

## 亮点

- The Aurora-vs-Whisper parameter comparison table in Section 5 (Cell 12) is precise and genuinely useful — it covers n_fft, hop, n_mels, fmin/fmax, log normalization style, and output shape transposition in one scannable reference.
- The three-test battery in Cell 11 (nominal 30 s, 5 s short pad, 60 s long truncate) is the right set of edge cases for a shape-alignment exercise.
- The cross-attention shape demo in Cell 8 concretely shows Q:(B,T_dec,d), K/V:(B,T_enc,d), attention weight:(B,T_dec,T_enc), which is the right mental model for a first encounter with cross-attention.
- The `aurora.audio.mel` import path is real and correct (`src/aurora/audio/mel.py` exports both `mel_spectrogram` and `power_to_db`).

## 问题列表

### 🔴 Critical

**[Accuracy]** — Cell 5 makes a factually wrong and self-contradictory claim about Whisper's encoder downsampling. The prose states "原论文实际用的 stride=2 × 2，总降采样 4 倍" (the original paper uses stride=2×2, 4× total downsampling), but then immediately shows the correct math: stride=1 gives 3000 frames, stride=2 gives 1500 — i.e. 2× total downsampling. The official `openai/whisper` encoder (`model.py`) is unambiguously `Conv1d(stride=1) → Conv1d(stride=2)`, producing 1500 output steps from 3000 input frames. The "4倍" figure will confuse any student who subsequently reads the Whisper source code.
**Fix:** Delete the "stride=2 × 2，总降采样 4 倍" clause. Replace with: "两层卷积总降采样为 2 倍：stride=1 保持 3000 帧，stride=2 压缩至 1500 帧。"

**[Course Continuity]** — Cell 14 (本课收束) ends with a fragment: "…` 加载的编码器，观察 cross-attention 权重，并与 Aurora 的 STFT 频谱图做对齐可视化。" There is a stray backtick mid-sentence and the clause has no grammatical subject — it appears to be the second half of a deleted or merged sentence. The paragraph describes two incompatible things: L71 (decoding strategies) and something about loading an encoder and doing alignment visualization, with no coherent join.
**Fix:** Delete the fragment after the L71 description. The 本课收束 should end cleanly after "…序列质量差异。"

### 🟠 Major

**[Exercise Design / Runability]** — Cell 10 uses `pass` as the stub body, so `whisper_preprocess` silently returns `None`. When Cell 11 runs `feat.shape`, Python raises `AttributeError: 'NoneType' object has no attribute 'shape'` — not `NotImplementedError`. Students will not recognize this as a "not yet implemented" signal, and the assert message never fires.
**Fix:** Replace `pass` with `raise NotImplementedError("whisper_preprocess: 按步骤1-4实现本函数")` as the sole body of the stub.

**[Accuracy]** — Cell 6 print says "Conv1 输出" but the variable `x` has already been overwritten by `conv2` by the time the print runs. The string prints `dummy_mel`'s original shape `(1, 80, 3000)` → `x.shape (1, 512, 1500)` but labels the first part as Conv1's output shape (it is actually the pre-convolution input shape). The intermediate Conv1 output `(1, 512, 3000)` is never shown.
**Fix:** Save the intermediate result: `x1 = torch.gelu(conv1(dummy_mel))`, `x2 = torch.gelu(conv2(x1))`. Then print `(1, 80, 3000) → Conv1 → {tuple(x1.shape)} → Conv2 → {tuple(x2.shape)}`.

**[Pedagogical Quality]** — The lesson title and header prominently advertise "多任务头" (multi-task heads) as a third topic, but the notebook contains no dedicated section on it. The multi-task output heads — transcription head, translation head, language-ID head, and timestamp head — are fundamental to what makes Whisper "Whisper" versus a vanilla seq2seq model, and are the architectural feature most distinct from earlier lessons. The decoder section (Cell 7) briefly lists the special tokens but does not explain how the output projection is shared across tasks, nor the training mixture that produces the multi-task capability.
**Fix:** Add a Section 4 (shifting the current exercise to Section 5) that explains the shared output projection layer, the four task heads, and the special-token conditioning mechanism. At minimum, show the token conditioning sequence `[SOT][lang][task][notimestamps]→content` with a concrete example and a diagram or code sketch of the output head.

**[Pedagogical Quality]** — No 学习目标 (learning objectives) section. Every other lesson in the course includes a numbered, concrete set of objectives. The absence is conspicuous here.
**Fix:** Add a 学习目标 cell after the intro, e.g.:
1. 理解 Whisper 80-dim log-Mel 输入的生成过程及与 Aurora DSP 的参数对齐
2. 描述编码器 Conv1d×2+Transformer 的形状变换链（3000→1500→(1500,d)）

4. 实现 `whisper_preprocess` 并通过三组形状断言
5. 对比 Aurora 默认参数与 Whisper 参数的帧数和归一化差异

### 🟡 Minor

**[Accuracy]** — Cell 13 expects `mel_whisper.shape` to show `(3000, 80)` (implied by the comment "约 (3000, 80)" in Cell 12). But Aurora's `mel_spectrogram` uses `frame_signal(center=True)`, which pads `n_fft//2 = 200` samples on each side, giving `padded_length = 480400`, and `n_frames = 1 + (480400 − 400) // 160 = 3001`. The print will show `帧数=3001`, not 3000. The "约" qualifier partially hedges this, but the student's whisper_preprocess would need to truncate 3001→3000, contradicting the intro formula `n_frames = n_samples / hop = 3000`.
**Fix:** Note explicitly that Aurora's center=True STFT gives 3001 frames for 30 s at hop=160, and that the truncation step in whisper_preprocess handles this 1-frame overage.

**[Exercise Design]** — Cell 11 only asserts shapes and dtypes; the normalization check is a print, not an assertion. A student who returns `torch.zeros(1, 80, 3000, dtype=torch.float32)` would pass all three asserts but have a completely wrong implementation.
**Fix:** Add: `assert feat.max() < -0.5, f"静音归一化后最大值应<-0.5，实际={feat.max():.3f}"` (silence after Whisper normalization maps to `(log10(1e-10) + 4) / 4 = (-10 + 4) / 4 = -1.5`).

**[Accuracy]** — Cell 8 defines `head_dim = d_model // n_heads = 64` and uses it as the attention scale, but Q, K, V have shape `(B, T, d_model=512)` — they are not split into heads. Scaling a full-d_model attention map by `head_dim**0.5 = 8` instead of `d_model**0.5 ≈ 22.6` produces incorrectly large logits in the demo. The illustration is conceptually inconsistent: either split the heads or scale by d_model.
**Fix:** Either add a comment noting "this is a single-head approximation; use sqrt(d_model) for full-dim Q/K/V" or split Q/K/V into heads before the matmul.
