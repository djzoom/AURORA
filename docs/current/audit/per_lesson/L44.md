## 问题列表

### 🔴 Critical

**[Accuracy / Terminology]** — "补零 (zero padding)" is used to describe reflect padding in two
places, teaching the wrong concept.

Cell 7 ends with: "补零（zero padding）让首尾帧也能被完整分析，是 librosa / Aurora 的默认行为."  
Cell 8's print statement says: `print(f"补零后长度: {len(x_padded)}")` — but `x_padded` was
created with `np.pad(x, pad, mode="reflect")`.

Reflect padding and zero padding are not the same: reflect padding mirrors signal content at
the edges and avoids the transient energy at the boundary that zero padding introduces. Aurora
explicitly uses `mode="reflect"` (confirmed in `stft.py` line 43). Calling it "zero padding" is
factually wrong and contradicts what the code demonstrates.

**Fix**: Change the closing sentence in Cell 7 to "反射填充（reflect padding）让首尾帧也能被完整分析".
Change the Cell 8 print to `print(f"反射填充后长度: {len(x_padded)}")`. Add a sentence
distinguishing reflect padding from zero padding (which would create an amplitude ramp artefact
at the boundary).

### 🟠 Major

**[Exercise Design — unlabelled final TODO]** — The stub's `return None` at the bottom (line
after the for-loop) is never marked as a TODO, but it must be changed to
`return np.stack(spectra)` for the function to produce any output. A student who faithfully
implements all five labelled TODOs (1–5) still gets back `None`, fails all assertions, and has
no hint of why.

Specifically, after completing TODO 4 (`spectra.append(np.fft.fft(f)[:n_bins])`), the for-loop
correctly populates `spectra`, but execution immediately falls through to `return None`. The
student must also replace `return None` with `return np.stack(spectra)` — but this line carries
no TODO annotation.

**Fix**: Replace `return None` at the end of the function with:
```python
    # ✏️ TODO 5: stack spectra into 2D array and return
    return None  # replace with: np.stack(spectra)
```
Or better, renumber TODO 5 to cover the stack-and-return, and label the guard return separately.

**[Runability — Cell 13 crashes before TODO is done]** — Cell 13 (visualization) calls
`np.abs(my_stft(x_chirp, ...)) ** 2`, but `my_stft` returns `None` until the student finishes
the exercise. `np.abs(None)` raises `TypeError`, breaking top-to-bottom execution for any
student reading the notebook before completing the TODO.

**Fix**: Add a guard at the top of Cell 13:
```python
if out is None:
    print("⚠️  my_stft 尚未实现，跳过可视化")
else:
    # ... rest of visualization
```
Or, restructure the notebook so the visualization cell appears after the verification cell (Cell
11) with an explicit note that Cell 13 requires the exercise to be done first.

---

**[Exercise Design — stub does not use NotImplementedError]** — The stub uses `win = None`,
`frames = None`, `windowed = None` as placeholders. This is a pattern that silently succeeds in
unexpected ways: `if windowed is None: return None` short-circuits the function body when only
TODOs 1-2 are done, but once TODO 3 is done (`windowed = frames * win`), the line fails with a
`TypeError` if `frames` is still `None`. Using `raise NotImplementedError` sentinels would give
students immediate, clear feedback about which step is missing.

**Fix**: Replace `win = None`, `frames = None`, `windowed = None` with
`raise NotImplementedError("TODO 1: generate window")` etc., or at minimum replace the
intermediate guard `if windowed is None: return None` with `assert windowed is not None` to
produce a meaningful error.

### 🟡 Minor

**[Pedagogy — conceptual gap: np.fft.fft vs aurora.audio.rfft]** — The exercise instructs
`np.fft.fft(f)[:n_bins]` (Cell 9 scaffold), but Aurora's `stft()` internally calls
`aurora.audio.transforms.rfft()`, which wraps the hand-written Cooley-Tukey FFT. The verification
test at `atol=1e-9` passes because the hand-written FFT is validated against numpy in the test
suite, but the notebook never explains this. Students may wonder why they are allowed to use
`np.fft.fft` when Aurora has its own FFT.

**Fix**: Add one sentence in Cell 9: "这里使用 `np.fft.fft` — Aurora 的手写 `rfft` 已通过
`tests/audio/test_transforms.py` 验证与 numpy 在 1e-9 以内一致，两者结果相同."

---

**[Pedagogy — n_fft vs win_len naming]** — Aurora's `stft()` signature uses `n_fft`, but
`my_stft()` uses `win_len`. For a power-of-two `n_fft`, these are equivalent, but the notebook
never explains that STFT can zero-pad frames to an `n_fft > win_len` for frequency interpolation.
The silent name difference will confuse students who then look at `stft.py`.

**Fix**: Add a comment in Cell 10's docstring: "`win_len` 与 Aurora 的 `n_fft` 等价（此处不做
额外零填充；如需插值可令 `n_fft > win_len`）."

---

**[Runability — Cell 4 prints formula as literal string]** — 
`print(f"n_frames = 1 + (padded_len - win_len) // hop")` prints the formula without evaluating
it. The student cannot verify whether their `frames.shape[0]` matches the formula.

**Fix**: 
```python
n_frames_formula = 1 + (len(x_padded) - win_len) // hop

```
