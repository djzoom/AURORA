### 🟠 Major

**[Accuracy]** cell-12 预期帧数严重错误。笔记本声称 win_len=128/hop=32 给出"约 125 帧"，win_len=2048/hop=512 给出"约 7 帧"，实际上对 N=4000 个采样点分别为 **122 帧**和 **4 帧**。后者差距接近 2 倍，会让学生以为 0.5 秒信号能切出 7 个宽达 256ms 的窗口。修复：将"约 125 帧"改为"约 122 帧"，"约 7 帧"改为"约 4 帧"。

**[Accuracy]** cell-12 帧数之比断言错误。"帧数之比约为 2048/128 = **16 倍**"基于大信号渐近近似；对 N=4000 实际运行结果是 122/4 = **30.5 倍**，偏差近一倍。cell-13 末尾的打印会让学生看到与预期完全不符的数字，无提示说明。修复：去掉固定的"16倍"断言，改为"（大信号时渐近 16 倍，本例 N=4000 时约 30 倍，因 2048 窗在 4000 点信号上只能完整切出 4 帧）"，或将 N 改为 88200（10 秒 sr=8000）使近似成立。

**[Accuracy]** cell-12 频率 bin 数全部差一。"64 个"应为 65 个，"1024 个"应为 1025 个——rfft 输出 win_len//2+1 而非 win_len//2。这与 cell-3 的输出形状定义 `(n_frames, win_len//2+1)` 自相矛盾，极易混淆学生。修复：将"freq_bins 少（64 个）"改为"freq_bins 少（65 个，即 win_len//2+1）"；"freq_bins 多（1024 个）"改为"1025 个"。

**[Runability]** cell-13 在 `frame_signal` 未实现时直接崩溃。`frames = frame_signal(x, win_len, hop)` 返回 `Ellipsis`，随后 `frames.shape[0]` 抛出 `AttributeError: 'ellipsis' object has no attribute 'shape'`。cell-11 已妥善处理此情形，cell-13 未跟进，导致笔记本在练习完成前无法从头到尾运行。修复：在 cell-13 顶部加一个守卫：

```python
if frame_signal(np.arange(10), 4, 2) is ...:
    print('⬜ 请先完成 frame_signal 练习'); raise SystemExit
```

或对整个 cell 使用 try/except AttributeError。
