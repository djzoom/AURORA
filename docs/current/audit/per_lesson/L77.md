## 概述

L77 follows a solid demo-then-exercise structure and the math for chroma and RMS is correct. However, the lesson contains a critical factual error — it claims `aurora.music` "尚未创建" (not yet created) when the subpackage is fully implemented at `src/aurora/music/` with `chroma_vector`, `rms_envelope`, `zero_crossing_rate`, `onset_envelope`, and `beat_track` all exported. On top of that, the title promises ZCR but the notebook covers BPM autocorrelation instead, and the chroma normalization taught (max-norm to [0,1]) silently diverges from the reference implementation (L1-norm). Together these issues mean a diligent student would finish the lesson holding a mental model that contradicts the actual codebase.

## 亮点

- The three-part progression — explain formula → working demo → scaffolded exercise — is pedagogically clean and mirrors the best lessons in the earlier DSP unit.
- The RMS parameter experiment (cell 16, comparing hop=128/512/2048) gives genuine insight into the resolution-smoothness tradeoff without extra verbosity.
- The autocorrelation BPM explainer (Section 3) is well-written: lag-to-BPM conversion formula is derived inline, the range bounds are shown explicitly, and the visual of the autocorrelation curve with a marked peak is informative.
- Assertion messages in test cells are descriptive (`f'期望 9 (A), 得到 {peak_pc}'`), which aids debugging.
- The solutions directory (`notebooks/8_music/solutions/L77_music_features_solutions.md`) actually exists, so the hint references are valid.

## 问题列表

### 🔴 Critical

**[Accuracy / Course Continuity]** — The "Aurora 连接" section states that `aurora.music` "尚未创建" and that the student is working ahead of the package. This is false. `src/aurora/music/__init__.py` already exports `chroma_vector`, `rms_envelope`, `zero_crossing_rate`, `onset_envelope`, and `beat_track`. Students who inspect the package will find complete reference implementations and become confused about the purpose of the exercises.

Fix: Replace the inaccurate claim with an accurate description — e.g., "本节练习将从零实现 `aurora.music.features` 中已有的 `chroma_vector` 和 `rms_envelope`，以理解其内部逻辑；实现完成后可与 `src/aurora/music/features.py` 的参考实现对比。"

### 🟠 Major

**[Accuracy — Normalization Mismatch]** — The exercise description (Section 4) instructs students to normalize chroma by dividing by the maximum value (max-norm, range [0,1]), and the test asserts `abs(ch[9] - 1.0) < 1e-6`. The reference implementation in `src/aurora/music/features.py` (`chroma_vector`) uses **L1 normalization** (`chroma /= chroma.sum()`). A student who follows the notebook will produce a function whose output differs from the library function for any multi-pitch input — with no warning of this divergence. The test assertion specifically only holds under max-norm.

Fix: Choose one normalization and apply it consistently. The lesson should either (a) adopt L1-norm to match the library and update the test threshold accordingly, or (b) explicitly acknowledge the two conventions and explain why the lesson uses max-norm as a pedagogical simplification, noting that the library uses L1-norm.

**[Exercise Design — Title/Content Mismatch, Missing ZCR]** — The lesson title is "chroma、RMS 能量、**ZCR**" but the notebook replaces ZCR with an autocorrelation BPM demo (Sections 3 and 8). ZCR (`zero_crossing_rate`) is fully implemented in `aurora.music.features` and is a natural fit here (it shares the same frame-hop structure as RMS). The student is never taught or asked to implement it, despite the title promising it.

Fix: Either (a) add a Section 6 exercise for `zero_crossing_rate` (analogous to the RMS exercise: formula → demo → stub → tests including a pure sine vs. noise comparison) and drop or move the BPM section, or (b) change the title to accurately reflect the content ("chroma、RMS 能量、节拍检测").

**[Runability — rms_envelope stub crashes test cell]** — The stub sets `n_frames = 0`, causing `frames = np.zeros((0, frame_len))` and returning `np.zeros(0)`. The first assertion in cell 14 then calls `rms_sil.max()` on an empty array, raising `ValueError: zero-size array to reduction operation fmax which has no identity`. The test cell crashes with a runtime exception rather than giving the student an informative AssertionError, masking the real feedback.

Fix: Change the stub to `raise NotImplementedError("rms_envelope: 请完成实现")` as the CLAUDE.md coding convention requires for substantive exercises, so the test cell fails early with a clear message instead of crashing midway.

### 🟡 Minor

**[Runability — Unused imports]** — `from aurora.audio.stft import stft` and `from aurora.audio.io import read_wav` are imported in cell 2 but never called. All spectral computations in this notebook use `np.fft.rfft` directly. While the imports succeed (both modules exist), they create dead weight and mislead students about what the notebook actually uses.

Fix: Remove both unused imports from cell 2. If a subsequent revision adds an exercise that calls `stft` (e.g., for chromagram), add the import back at that point.

**[Exercise Design — No numbered learning objectives]** — The "目标" paragraph states a goal in prose but is not formatted as a numbered list of specific, testable learning objectives (e.g., "1. 能够描述 chroma 向量如何将 FFT 频率箱映射到音高类..."). The rest of the course (e.g., L76, L78) uses structured objectives.

Fix: Replace the single-paragraph goal with a numbered 学习目标 list of 3–4 concrete objectives aligned to the exercises and demo sections.

**[Exercise Design — chroma_vector stub should raise NotImplementedError]** — The `chroma_vector` stub sets `midi = 0.0` and executes `pass`, silently accumulating nothing. The test then sees `ch.argmax() = 0` (first zero) and raises `AssertionError: 期望 9 (A), 得到 0`. This is acceptable but less informative than a `raise NotImplementedError` which signals "not implemented yet" rather than "wrong answer." Per CLAUDE.md: stubs should `raise NotImplementedError`.

Fix: Replace the stub body with `raise NotImplementedError("chroma_vector: 请实现该函数")`.

**[Accuracy — BPM search range is off-by-one (excludes 60 BPM)]** — `lag_max = int(60/60 * sr / hop) = 43`. The slice `ac[lag_min:lag_max]` checks lags 10–42 inclusive, meaning the lowest BPM reachable is `60 / (42 * 512 / 22050) ≈ 61.5`. A signal at exactly 60 BPM is excluded. The lag_min/lag_max comment says "60–240 BPM" but the actual range is ~61.5–240 BPM.

Fix: Change `lag_max = int(60/60 * sr / hop) + 1` so the slice includes lag=43 and reaches exactly 60 BPM.

**[Accuracy — `np.fft.rfft` used directly in demo and test cells]** — Cells 4 and 11 use `np.fft.rfft` directly rather than calling `aurora.audio.stft.stft`. While importing `stft` at the top signals intent to use it, the actual demo code bypasses it. This is inconsistent with the project's "no black-box wrappers" philosophy (even though `aurora.audio.stft` wraps its own hand-written `rfft`, using it here would keep the demo within the course's own stack and reinforce the API).

Fix: Replace the direct `np.fft.rfft` calls in cells 4 and 11 with `aurora.audio.stft.power_spectrogram(x, n_fft=n_fft, hop_length=n_fft)` or compute the single-frame spectrum via `aurora.audio.stft.stft`, consistent with how the library is structured.
