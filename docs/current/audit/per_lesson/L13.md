## 概述

L13 covers three important matrix classes with accurate core mathematics and a sensible hands-on exercise for `is_orthogonal`. The FFT roundtrip verification cell is a genuine pedagogical highlight — students prove the unitary property without waiting until L39. However, the notebook contains one critical factual error: matrix `[[4., -2.],[-2., 1.]]` is labelled "对称（不定）" but its eigenvalues are 0 and 5, making it semi-positive definite, not indefinite — directly contradicting what a student computes. Three major issues compound this: the exercise stub returns `None` (causing a silent assertion failure), `dft_matrix(n)` is cited three times as a real Aurora function but exists nowhere in `src/aurora/`, and the Positive Definite section has no definition, no exercise, and no Aurora connection despite being a title promise.

- Cell 22 (`ifft(fft(x)) == x` roundtrip with `atol=1e-12`) is concise, runnable, and self-motivating. The L39 forward reference is accurate: `notebooks/5_audio_dsp/L39_fft_implement.ipynb` exists and covers Cooley-Tukey.

## 问题列表

### 🔴 Critical

**[Accuracy]** — Cell 9 (cell-8): matrix `np.array([[4., -2.],[-2., 1.]])` is annotated `# 对称（不定）`, but its eigenvalues are λ = 0 and λ = 5 (trace = 5, det = 4·1 − 4 = 0). A zero eigenvalue makes it **semi-positive definite**, not indefinite. Indefinite matrices require at least one positive and one negative eigenvalue. A student runs this cell, sees `[0. 5.]`, reads "不定", and builds the wrong mental model.
Fix: Replace the matrix with a genuinely indefinite example, e.g., `np.array([[1., 2.],[2., 1.]])` (eigenvalues −1 and 3), or relabel the annotation to "对称（半正定）".

### 🟠 Major

**[Exercise Design]** — Cell 14 (2cdd8d3f): the `is_orthogonal` stub body is `return None` instead of `raise NotImplementedError`. In Cell 15 the assert fires as `assert None and ...` which short-circuits to a bare `AssertionError()` with no message. The printed lines show `Q 正交? None (应 True)`, which is confusing: students cannot tell whether the code itself is broken or they simply haven't implemented the function yet.
Fix: Change stub to `raise NotImplementedError("implement is_orthogonal: return True iff Q.T @ Q ≈ I")`. Add messages to the asserts: `assert is_orthogonal(Q), "Q (90° rotation) must be orthogonal"`.

**[Accuracy / Aurora 连接]** — `dft_matrix(n)` is cited three times as a real Aurora package function: in Learning Objective 5, in "为什么对 Aurora 重要", and in "本课收束". The file `src/aurora/audio/dft.py` does not exist; `src/aurora/audio/transforms.py` defines `dft()` and `idft()` but no `dft_matrix()`. No notebook defines this function as importable from `aurora`. Students who search the codebase or attempt `from aurora.audio.transforms import dft_matrix` get `ImportError`.
Fix: Replace references with `aurora.audio.transforms.dft()` (which exists), or implement `dft_matrix` in `transforms.py` and export it. Add a clarifying note that the matrix-form view is developed inline in L21, not as a package function.

**[Pedagogical Quality]** — Positive definiteness is one of the three topics promised in the title, but it receives only a standalone code demo (Cell 19) with no Markdown definition, no explanation of why Cholesky success implies PD, and no TODO exercise. The algebraic definition (`xᵀAx > 0` for all nonzero x, equivalent to all eigenvalues > 0) never appears. The Aurora connection (e.g., covariance matrices in PCA must be PSD) is absent. The treatment is shallow compared to the orthogonal matrix section.
Fix: Add a Markdown cell before Cell 19 with the PD definition and the Cholesky criterion. Add a small exercise: construct a 3×3 PD matrix via `A = M.T @ M` and verify with `np.linalg.cholesky`.

### 🟡 Minor

**[Runability]** — Cell 11 (a770d882) uses `np.array`, `np.cos`, `np.sin`, and `np.linalg.norm` with no `import numpy as np`. It relies on the import in Cell 7. Running Cell 11 after a kernel restart or in isolation gives `NameError: name 'np' is not defined`.
Fix: Add `import numpy as np` at the top of Cell 11.

**[Exercise Design]** — The `is_orthogonal` test in Cell 15 covers only one positive case (90° rotation, 2×2) and one negative case. Missing: `np.eye(3)` (trivial orthogonal), a 3×3 Householder reflection (det = −1 orthogonal), and `np.diag([2.0, 1.0])` (scaling, not orthogonal). The spec also does not state expected behavior for non-square input; the reference solution uses `n = len(Q)` which returns the row count and would compare against the wrong-sized identity for a tall matrix.
Fix: Add three additional asserts. Document that the function is defined only for square matrices.

**[Course Continuity]** — Three cells give conflicting signals about when `ifft(fft(x)) == x` is verified. Cell 16 says "这就是 L39 你会验证的事"; Cell 21 heading says "当场验证 ifft(fft(x)) == x" and Cell 22 prints "不用等 L39"; Cell 23 says "L39 验证 ifft(fft(x)) == x 时直接依赖这条性质." The mixed messaging is confusing.
Fix: Standardize: L13 verifies the property with numpy; L39 re-derives it from the Aurora hand-implemented DFT matrix. Remove the contradictory "不用等 L39" framing since it undermines L39's motivation.

**[Pedagogical Quality]** — The title metaphor "对称矩阵 = 镜子" appears only in the title and nowhere in the body. Students may confuse symmetric matrices with reflection matrices (orthogonal matrices with det = −1), which are genuinely "mirrors" geometrically. The notebook never resolves this potential confusion.
Fix: Either drop the "镜子" metaphor or add one sentence explaining the distinction: "对称矩阵 Aᵀ = A 的"镜像"指矩阵元素关于主对角线对称，与几何反射矩阵（det Q = −1 的正交矩阵）不同."

**[Accuracy]** — The reference solution in `solutions/L13_special_matrices_solutions.md` uses `n = len(Q)` to build `np.eye(n)`. For non-square Q with shape (m, k), `Q.T @ Q` is (k, k) but `np.eye(len(Q))` is (m, m). The shapes mismatch and `np.allclose` raises `ValueError`. The solution needs `Q.shape[1]` (column count) to be correct.
Fix: Change `n = len(Q)` to `n = Q.shape[1]` in the reference solution and add a guard against non-2D input.
