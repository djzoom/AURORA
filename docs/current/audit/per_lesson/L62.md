# L62 — Dataset 与 DataLoader — 自定义 __getitem__，音频数据批量加载

**总体评级**: B- | **问题数**: 1 critical / 3 major / 5 minor

## 概述

本节结构清晰，pad/truncate、per-sample 归一化的讲解简洁准确，`extract_features` 练习的脚手架设计合理（`raise NotImplementedError`、分步 TODO、指向 solutions 目录）。然而 `KWSDataset.__getitem__` 存在两处相互叠加的实现错误：缺少转置且未传递 `n_fft`/`hop_length`，导致其输出形状 `(T=63, n_mels=40)` 与文档承诺的 `(n_mels=40, T=32)` 完全不同，直接下游的 L63 CNN 会立即崩溃。`power_to_db` 的不一致进一步造成两条特征提取路径在数值上无法对齐。

## 亮点

- `pad_or_truncate` 实现简洁正确，演示了边界行为（短/长两种输入），注释解释了"补零不影响分类器"的原因，教学价值高。
- Per-sample 归一化与 BatchNorm 的对比说明清晰，解释了为何此处选用 per-sample 策略，符合课程深度。
- `extract_features` 练习的六步推理路线（TODO 1–6）清晰，学生可以逐步实现，有 `raise NotImplementedError` 阻止提前通过。
- Aurora 连接明确且真实：`aurora.audio.mel.mel_spectrogram` 确实存在于 `src/aurora/audio/mel.py`，`aurora.audio.io.read_wav` 存在且自动下混立体声。

## 问题列表

### 🔴 Critical

**[代码准确性 / 可运行性]** — `KWSDataset.__getitem__` 缺少转置，且使用错误的默认参数，输出形状完全错误。

Cell 15 的实现：
```python
mel = mel_spectrogram(x, sr, n_mels=self.n_mels)
feat = torch.tensor(normalize(power_to_db(mel)), dtype=torch.float32)
```
- `mel_spectrogram` 返回 `(n_frames, n_mels)` = `(T, 40)`（aurora 约定：时间在前），但缺少 `.T` 转置，`feat` 形状为 `(T, 40)` 而非 `(40, T)`。
- 没有传递 `n_fft=2048, hop_length=512`，aurora 默认 `n_fft=1024, hop_length=256`（`stft.py` 中 `hop_length = n_fft // 4`）。实际计算：`T = 1 + (16000 // 256) = 63`，不是文档承诺的 32。
- `DataLoader` 组成的批次形状将是 `(batch, 63, 40)`，而 L63 CNN 期望 `(batch, 40, T)`。
- 验证断言 `batch_x.ndim == 3` 无法发现此错误。

**Fix**: 在 `mel_spectrogram` 调用后加 `.T`（或用 numpy 转置），并显式传递 `n_fft=2048, hop_length=512`：
```python
mel = mel_spectrogram(x, sr, n_mels=self.n_mels, n_fft=2048, hop_length=512)
mel = mel.T  # (n_mels, T)
feat = torch.tensor(normalize(power_to_db(mel)), dtype=torch.float32)
```
同时将 cell 15 的断言增强为：
```python
assert batch_x.shape[1] == 40, f"dim 1 应为 n_mels=40，得 {batch_x.shape[1]}"
assert batch_x.shape[2] > 0   # T 帧数
```

### 🟠 Major

**[代码准确性]** — `power_to_db` 在 KWSDataset 中被应用，但在 `extract_features` 规格（cell 9）和 Section 3 归一化说明中均未提及，造成两条特征提取路径数值不一致。

学生按 cell 9 的六步 TODO 实现 `extract_features`（mel → 归一化，无 dB 转换），而 `KWSDataset.__getitem__` 则做 mel → dB → 归一化。两者馈入同一个 L63 分类器但特征量纲不同（功率 vs dB），性能差异无法解释。

**Fix**: 在两条路径中统一决策（推荐加 dB 转换，因为对数压缩更符合感知且对 CNN 训练更稳定）。同步更新 cell 9 的 TODO 步骤，在步骤 4–5 之间增加步骤 4b：`mel_db = power_to_db(mel_T)`。

---

**[代码准确性]** — cell 11 的断言仅检查 `feat.shape[0] == 40`，不检查时间维度 T。

如果学生实现了错误的 hop_length（例如用默认的 256），`shape[0]` 仍为 40（n_mels），断言通过，但特征与下游 CNN 的期望不兼容。

**Fix**:
```python
assert feat.shape == (40, 32), f"期望 (40, 32)，得 {tuple(feat.shape)}"
assert feat.dtype == torch.float32, "dtype 应为 float32"
```

---

**[代码准确性]** — `label_map.get(label, len(label_map))` 对未知标签返回越界索引。

若 label_map 有 4 个类别（indices 0–3），`len(label_map)=4` 作为默认值会在 CrossEntropyLoss 或 embedding 层引发 index out of range，且无错误提示指向原因。

**Fix**: 改为报错或映射到合法的"unknown"类别：
```python
if label not in self.label_map:
    raise ValueError(f"未知标签 '{label}'，合法标签：{list(self.label_map)}")
label_idx = self.label_map[label]
```

---

### 🟡 Minor

**[数学准确性]** — Cell 9 注释写 "T = ceil(16000/512) ≈ 32"，用 "≈" 不准确。

实际计算（aurora 使用 `center=True`，pad = n_fft//2 = 1024）：T = 1 + (16000 // 512) = 32，精确等于 32。"≈" 应改为 "="，并说明公式来源：`T = 1 + ⌊n_samples / hop_length⌋`（center 模式下）。

---

**[可运行性]** — cells 11 和 13 均使用 `NamedTemporaryFile(delete=False)` 而不清理，会在学生机器上泄漏临时文件。

**Fix**: 使用 context manager 或在 finally 块中调用 `os.unlink(tmp.name)`：
```python
import os
...
os.unlink(tmp.name)
```

**[可运行性]** — 重复导入：cell 13 重复了 cell 11 中的 `import tempfile, soundfile as sf`；cell 15 重复了 cell 2 中已存在的 `import numpy as np, torch`。

**Fix**: 将所有顶层导入集中到 cell 2，cell 13 和 15 的重复导入行删除。

**[练习设计]** — Cell 10 的 TODO 步骤 2 提示"处理立体声"，但 `aurora.audio.io.read_wav` 已在内部自动将多声道下混为单声道（`data.reshape(-1, n_channels).mean(axis=1)`）。学生无需额外处理，但提示会使其困惑或添加多余代码。

**Fix**: 将 TODO 2 改为"断言 `file_sr == sr`（数据集均为 16 kHz，直接断言即可）"，删除"处理立体声"措辞。

**[练习设计]** — `x, file_sr = None, None` 初始化桩（cell 10）会造成若学生忘记实现 TODO 1 而继续执行后续步骤时产生 `AttributeError: 'NoneType' object has no attribute ...` 而非更清晰的错误。

**Fix**: 删除该初始化行，让 `read_wav` 的调用成为学生的第一步；或换成：
```python

```

## 优先修复（前3项）

1. **[Critical] 修复 KWSDataset.__getitem__ 的形状错误**：加转置（`.T`）并显式传 `n_fft=2048, hop_length=512`，将输出从 `(63, 40)` 修正为 `(40, 32)`，否则 L63 CNN 无法运行。

2. **[Major] 统一 power_to_db 的使用**：在 cell 9 的六步 TODO 中增加 dB 转换步骤，与 KWSDataset 路径对齐，确保两条特征提取路径输出数值一致。

3. **[Major] 加强 cell 11 的形状断言**：改为 `assert feat.shape == (40, 32)`，同时覆盖 n_mels 和 T 两个维度，防止学生因参数错误而自信地通过检验。
