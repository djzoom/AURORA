L72 的数学和 LoRA 理论部分写得清晰准确，参数量推导自洽，WER 单元测试也是一个好习惯。然而，课程题目中最重要的承诺——"中文 / 方言数据适配实战"——在正文里完全缺席，整节课只讲英语 LibriSpeech；同时有一个会在运行时直接抛异常的数据集分割名错误，以及对 `aurora.audio.mel` 默认参数的误导性描述。对于一节以"实战"命名的课，没有任何学生 TODO 练习也是结构性缺陷。

## 亮点

- **LoRA 参数量推导具体且正确**：notebook 从 `d_model=768, r=8` 出发，算出原始 q/v 参数 ≈1.18M，LoRA 参数 24,576，降幅 98%，数字完全自洽，可作为学生手算的参考范本。
- **WER 单元测试**：在 `compute_metrics` 定义完成后立即用一个已知答案（1 次替换，6 词，WER=1/6）做验证，assert 精度合理（`< 1e-6`），是教学中值得推广的即测即用风格。

## 问题列表

### 🔴 Critical

**[Title/Content Mismatch]** — 课程标题明确承诺"中文 / 方言数据适配实战"，但正文中完全没有中文或方言相关内容。所有示例均使用英语 LibriSpeech + 英语 WhisperProcessor（`language="English"`），没有任何一处涉及 Common Voice zh-CN、AISHELL-1 或方言数据的处理差异（分词器语言切换、字符级 WER、Whisper 中文提示词等）。标题是实质性的虚假承诺。
**Fix**：要么增加一个完整的"中文适配"章节（至少演示语言切换到 `language="Chinese"`、使用中文字符级 WER、以及数据集替换为 AISHELL-1 或 Common Voice zh），要么修改课程标题去掉这一承诺。

**[Runability — Runtime Error]** — 数据集加载代码使用了不存在的 split 名称：
```python
split={"train": "train.clean.100", "test": "test"}
```
LibriSpeech 在 HuggingFace 上没有名为 `"test"` 的 split；正确名称是 `"test.clean"`（或 `"test.other"`）。此处将在运行时抛出 `ValueError`，整个数据管道无法启动。
**Fix**：将 `"test"` 改为 `"test.clean"`。

**[Accuracy — Aurora 连接虚假]** — Cell-0 声称 `aurora.audio.mel.mel_spectrogram` "均采用 25 ms 帧长、10 ms 步长"，但查阅 `src/aurora/audio/mel.py` + `stft.py` 可见其默认值是 `n_fft=1024`（在 16 kHz 下为 64 ms）和 `hop_length=n_fft//4=256`（16 ms）。Whisper 的 25 ms / 10 ms 对应 `n_fft=400, hop_length=160`，需要显式传参才能对齐。用默认参数调用 aurora 的函数会产生与 Whisper 不同的特征，导致"复用于离线特征缓存"的说法失效。
**Fix**：将 Aurora 连接描述改为"调用 `mel_spectrogram(signal, sr, n_fft=400, hop_length=160, n_mels=80)` 可与 Whisper 预处理对齐"，并说明默认参数不同。

### 🟠 Major

**[Pedagogical Quality — 无学习目标]** — Notebook 缺少标准的"学习目标"（学习目标）列表。对于第 72 节这样覆盖 LoRA 注入、Seq2SeqTrainer、WER 计算三个独立主题的课，明确的编号目标是必要的导航锚点。
**Fix**：在 Cell-0 后增加带编号的学习目标，例如：①掌握 LoRA 低秩分解原理及参数量计算；②能用 `peft` 注入 LoRA 并解读 `print_trainable_parameters` 输出；③配置 `Seq2SeqTrainer` 并理解各关键参数；④用 jiwer 计算 WER 并解释 S/D/I 含义。

**[Exercise Design — "实战"课无练习]** — 标题和定位是"实战"，但全课无任何 TODO 练习，所有代码均为演示性代码块（demonstration cells），学生没有主动参与机会。对比 L73，它有一个完整的 `analyze_errors()` TODO 框架（含 NotImplementedError 级别的 stub 和 assert 验证）。
**Fix**：至少增加一个练习，例如：让学生手动计算给定 LoRA rank 下的参数量（填写公式），或实现一个最简 WER 计算函数（不依赖 jiwer），或用不同 `r` 值重新配置 LoRA config 并观察 `print_trainable_parameters` 变化。

**[Course Continuity — WER 内容与 L73 重叠]** — L72 Section 3 引入了 WER 公式 `(S+D+I)/N`、jiwer 计算、正则化归一化，并做了单元测试。L73 Cell-0/Cell-1/Cell-3 立即重新介绍相同的公式和 S/D/I 定义，且用完全一致的文字。两节课之间没有清晰的知识边界。
**Fix**：L72 的 WER 部分缩减为"评估指标背景"（仅给出公式，不做 jiwer 演示），将具体实现和测试下推给 L73；或者在 L73 开头明确写"我们在 L72 已经看到 WER 定义，本节聚焦拆解 S/D/I 并实现编辑回溯"。

**[Runability — API 弃用导致警告/错误]** — `Seq2SeqTrainingArguments` 中使用了 `evaluation_strategy="steps"`，该参数在 `transformers >= 4.38` 中已重命名为 `eval_strategy`，使用旧名会触发 `FutureWarning` 并在部分版本中被拒绝。同样，`Seq2SeqTrainer(tokenizer=...)` 参数在较新版本中已弃用，建议改为 `processing_class`。
**Fix**：将 `evaluation_strategy` 改为 `eval_strategy`；将 `tokenizer=processor.tokenizer` 改为 `processing_class=processor`（或保留旧参数并在注释中说明版本依赖）。

### 🟡 Minor

**[Runability — 重复 import]** — Cell-10 中有 `import torch`，而 Cell-6 已经 `import torch`。在顺序执行时无害，但在重置 kernel 后单独运行 Cell-10 仍会因缺少 `model`、`training_args` 等而失败；重复 import 也是编码规范问题。
**Fix**：删除 Cell-10 中多余的 `import torch`。

**[Runability — BasicTextNormalizer 导入路径不稳定]** — `from transformers.models.whisper.english_normalizer import BasicTextNormalizer` 是一个私有/内部路径，在不同版本的 transformers 中路径有变动（如 4.39+ 中该模块位置调整）。
**Fix**：改用更稳定的公共导入或在 try/except 中包一层，并注明所依赖的 transformers 最低版本。

**[Pedagogical Quality — 训练曲线完全注释]** — Section 2 末尾绘制 loss / WER 曲线的代码整块被注释，学生无法看到训练过程的可视化。这是理解过拟合最直观的工具，应在有 GPU 的情况下自动绘制（或至少在 CPU 路径中用模拟数据演示曲线形状）。
**Fix**：取消注释，或将其放入 `if torch.cuda.is_available():` 块内，保证在 GPU 环境中自动执行。

**[Runability — trust_remote_code 未说明]** — `load_dataset(..., trust_remote_code=True)` 是一个安全敏感参数，允许执行 HuggingFace 数据集脚本中的任意代码，但 notebook 未解释为何需要它。
**Fix**：添加一行注释说明为何对 `librispeech_asr` 需要此参数，提醒学生不要对未知数据集盲目启用。

**[Accuracy — LoRA 初始化描述不完整]** — Section 1 说 `A ~ N(0, σ²), B = 0`，但原始 LoRA 论文（Hu et al. 2022）是 `A ~ N(0, 1/r)`（即 σ² = 1/r），而 peft 库的实现默认使用 kaiming_uniform。此处 `σ²` 没有具体数值，造成概念模糊。
**Fix**：明确说明 peft 的实际默认初始化策略，或直接引用"A 用 Kaiming uniform 初始化，B=0"这一实现级描述。
