## 概述

L48 is a purely visual demonstration lesson that correctly walks students through framing, the Mel filterbank geometry, and the three-way linear/Mel/log-Mel spectrogram comparison on a chirp signal. The overall structure is coherent and the plotting code runs correctly end-to-end. However, two factual statements in the prose are wrong relative to the actual `aurora.audio` implementation — the Hann window formula shown is the symmetric variant while the STFT uses the periodic (DFT-even) variant, and the filterbank is described as "area-normalized" when it is not — and the dB-conversion asymmetry (`20·log10` vs `10·log10`) is left unexplained. For a course centered on first-principles accuracy these are meaningful errors.

## 亮点

- The framing animation (cell 4) is an effective visualization; using `fill_between` on windowed frames with per-frame color is pedagogically clear.
- The two-panel filterbank plot (cell 6) — linear-frequency axis vs Mel axis — precisely illustrates the "low-frequency dense, high-frequency sparse" asymmetry and is the strongest cell in the notebook.
- The chirp experiment (cells 7–8) is well-chosen: a linear frequency sweep makes the contrast between linear and Mel axes immediately visible and memorable.
- The "本课收束" correctly names output shapes, correctly references `aurora.audio.stft` and `aurora.audio.mel`, and correctly points forward to L49 (MFCC/DCT).

### 🟠 Major

**[Accuracy] — Hann window formula mismatch with implementation.**  
Cell 3 states `w[n] = 0.5 · (1 − cos(2π n / (N−1)))` (symmetric form) and cell 4's visualization code uses the same `(n_fft - 1)` denominator. However, `aurora.audio.stft` calls `get_window(..., periodic=True)`, which uses denominator `N` (the DFT-even / periodic form, per the docstring in `windows.py`). The formula shown is not what AURORA computes. Fix: change cell 3 to `w[n] = 0.5 · (1 − cos(2π n / N))` and update the framing-demo code in cell 4 to `np.cos(2 * np.pi * np.arange(n_fft) / n_fft)`.

**[Accuracy] — False claim of area normalization in filterbank.**  
Cell 5 states: "面积归一化使各滤波器能量可比". The actual `mel_filterbank` in `src/aurora/audio/mel.py` (lines 63–69) clips `min(rising, falling)` to [0, 1] without dividing by bin width; triangles peak at 1.0 regardless of width. This is the HTK (unnormalized) convention. Higher-frequency filters span wider bands and collect proportionally more energy — the exact property that normalization would fix. Fix: either implement Slaney-style area normalization (`fb[m] /= (right - left)`) and document it, or remove the area-normalization claim and note the HTK convention explicitly.

**[Pedagogy] — dB conversion factor difference left unexplained.**  
Cell 8 uses `20 * np.log10(...)` for the amplitude spectrogram `S` and `10 * np.log10(...)` for the power mel spectrogram `M`, without any comment. Both colorbars are labeled "dB". Students will either assume it is a typo or not notice the difference. The two conversions are numerically consistent (`10·log10(|X|²) = 20·log10(|X|)`), but the distinction between amplitude-domain and power-domain quantities — and why each gets a different factor — is core DSP literacy. Fix: add a one-line comment: `# magnitude → 20·log10 ; mel_spectrogram returns power → 10·log10`.

### 🟡 Minor

**[Accuracy] — M_db transpose comment is wrong.**  
`M_db = (10 * np.log10(M + 1e-8)).T  # (n_frames, n_mels) → (n_frames, n_mels)` — the `.T` changes the shape to `(n_mels, n_frames)`. The comment claims the shape is unchanged, which is the opposite of what `.T` does. Fix: `# (n_frames, n_mels) → (n_mels, n_frames) for pcolormesh`.

**[Runability] — `stft` imported but never used.**  
Cell 2 imports `stft` from `aurora.audio.stft`; only `magnitude_spectrogram` is called in the notebook body. Fix: remove `stft` from the import line, or add a comment that it is available for student exploration.

**[Consistency] — `n_mels` differs between sections without explanation.**  
Section 2 (filterbank plot) uses `n_mels=40`; Section 3 (spectrogram) uses `n_mels=80`. No comment explains the change. A student who runs both cells back-to-back and inspects the filterbank shape will notice the inconsistency. Fix: use a single value throughout, or explicitly note "here we use 80 for higher frequency resolution."

**[Runability] — `sr` redefined silently.**  
`sr = 8000` in Section 1, `sr = 22050` in Sections 2–3 with no warning. Fix: add a cell comment `# switch to 22050 Hz for the main spectrogram demo` or scope the 8000 Hz value to a local variable.

**[Clarity] — `times_mel` is a redundant copy of `times`.**  
Both are `np.arange(n_frames) * hop_length / sr` computed from the same parameters. Fix: reuse `times`.

**[Pedagogy] — No structured 学习目标 section and no exercises.**  
The header "目标" is a single prose sentence. For a 99-lesson university course, even a purely visual lesson benefits from a numbered objective list (e.g., "能解释 Mel 轴为何低频密高频疏") and at least one exploratory exercise (e.g., "修改 hop_length 从 256 到 512，观察时间轴分辨率变化"). Fix: add a 学习目标 block with 3–4 numbered items and one "思考题" asking students to observe parameter effects.
