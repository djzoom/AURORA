## 问题列表

### 🔴 Critical

**[Accuracy]** — Cell 1 复杂度描述内部矛盾。原文声称"无缓存 O(t²) work per step → O(T³) total"、"有缓存 O(t) work per step → O(T²) total"，随即又说"两者 total FLOPs 量级相同"。O(T³) ≠ O(T²)，三句话自相矛盾。  
事实：每步 t，无缓存的 K/V 投影代价是 O(t·d²)（线性于 t，不是 t²）；attention 计算两者均为 O(t·d)；KV-Cache 节省的是**投影**部分，投影总代价从 O(T²·d²) 降至 O(T·d²)——即总投影量是 O(T²)→O(T)，而不是 O(T³)→O(T²)。标题"O(seq²)→O(seq)"指的是所有步的投影累积代价，应在正文中明确说清楚，而不是用"per step"措辞混淆。  
Fix: 重写 Cell 1，分别说明"每步投影代价"和"全序列投影总代价"，删除"两者量级相同"这句自相矛盾的话。

**[Runability/Accuracy]** — Cell 6 中 `KVCache(n_heads=4, head_dim=64)` 与实际数据 n_heads=2, head_dim=8 不匹配，且验证单元没有任何 assert 语句——`print('✅ KV-Cache 实现验证通过')` 是无条件打印，任何不崩溃的实现都会"通过"。  
Fix: 将 `KVCache(n_heads=2, head_dim=8)` 与数据对齐；加入显式 assert，例如断言 `cache.seq_len() == total_seq`、`out_with_cache.shape == (n_heads, total_seq, d)`，并对最终 token 的 attention 输出做数值一致性断言（正确的因果掩码版全量 attention vs 缓存版逐步 attention 最后 token 输出应相同）。

### 🟠 Major

**[Exercise Design]** — Cell 3 的 TODO 提示 `# scores = Q @ K^T / sqrt(head_dim)` 完全遗漏了 softmax 步骤，而 softmax 是 attention 的核心操作。学生按提示实现只会得到未归一化的分数矩阵乘以 V，结果完全错误。  
Fix: 将提示扩展为分步骨架：(1) `scores = ...`，(2) `if mask: scores[mask] = -inf`，(3) `weights = softmax(scores, axis=-1)`，(4) `return weights @ V`。

**[Exercise Design]** — Cell 3 验证只检查 `out.shape`，用 `return np.zeros_like(Q)` 就能通过。需要增加数值正确性断言：例如用已知输入的小规模手算结果做参考，或断言 attention 权重行之和为 1（softmax 性质）。  
Fix: 加入 `weights = Q @ K.transpose(0,2,1) / np.sqrt(d); weights = np.exp(weights) / np.exp(weights).sum(-1, keepdims=True); ref = weights @ V; assert np.allclose(out, ref, atol=1e-6)`。

**[Exercise Design]** — Cell 5 `update` 方法的 TODO 没有给出首次调用（layer 不在 `_k` 中）的任何提示。这是实现中最容易出错的边界情况，应显式引导。  
Fix: 在 `raise NotImplementedError` 前加注释骨架：`# 提示：if layer not in self._k: 直接赋值；else: np.concatenate(..., axis=1)`。

### 🟡 Minor

**[Runability]** — Cell 8 的计时循环内部调用 `np.random.randn` 生成数据，随机数生成本身的开销会混入计时结果，污染速度对比数据，尤其在小 T 值时噪声占比显著。  
Fix: 在计时循环外预先生成所有数据 `xs = [np.random.randn(t+1, d_model) for t in range(T)]`，计时循环内只做矩阵乘。

**[Exercise Design]** — `scaled_dot_product_attention` 的 `mask` 参数在整个 notebook 中从未被测试或实际使用。既然签名里有 mask，Cell 3 的验证应至少包含一个因果 mask 测试用例（如上三角 mask）。

**[Course Continuity]** — Cell 6 末尾注释"注意：无缓存版的 Q_all 包含所有 token 的 attention（非因果）...数值可能不同"承认了两种计算语义不同，但没有修正——这让验证的意图模糊，学生不知道他们的实现是否正确。应设计一个语义一致的测试（无缓存因果 vs 有缓存逐步）。
