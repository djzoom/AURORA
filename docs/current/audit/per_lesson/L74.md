# L74 — ASR 错误分析 — 替换/删除/插入模式，从 WER 到可改进方向

**总体评级**: B- | **问题数**: 2 critical / 3 major / 3 minor

## 概述

L74 presents a self-contained, technically correct WER decomposition demo: the edit-distance alignment and WER functions work, the S/D/I counters are accurate, and the lesson narrative connecting error types to improvement directions is sensible. However, the notebook reads as a pure demonstration rather than an interactive lesson — there are zero student exercises (no TODO stubs, no NotImplementedError) — which is the dominant failure at this point in a 99-lesson course. The "本课收束" section also contains two factual inaccuracies: it references a non-existent `aurora.asr` module and cites a function (`wer_decompose`) that is never defined anywhere in the notebook.

## 亮点

- The alignment backtracking correctly handles all boundary cases (i==0 or j==0), producing well-formed (op, ref_word, hyp_word) tuples.
- The 10 PAIRS cover a genuinely diverse range of ASR failure modes (digit/word confusion, OOV proper nouns, homophone substitution, sentence-final deletion), making the error pattern table realistic and discussion-ready.
- The WER decomposition formula `WER = (S + D + I) / N_ref` and its corpus-level validation in cell-9 are mathematically correct.
- The mapping from each error type to a concrete improvement direction (larger acoustic model / VAD tuning / noise suppression / hotword boosting) is pedagogically valuable and would hold up in a research-engineer interview.

## 问题列表

### 🔴 Critical

**[Exercise Design]** — The notebook contains zero student exercises. Every function (`alignment`, `wer`, the counters) is pre-implemented and ready to run. There are no `raise NotImplementedError` stubs, no TODO comments, and no verification cells that guard against a missing implementation. At L74 in a 99-lesson course, students should be implementing at minimum one of: (a) a confusion-matrix builder from the alignment output, (b) per-category WER breakdown (digit words vs. common words), or (c) a `wer_decompose` wrapper that returns a named-tuple with S/D/I counts. Fix: Add 1–2 scaffolded TODO exercises with clear specs, reasoning hints, and NotImplementedError stubs, then move the worked solution to `notebooks/7_asr/solutions/`.

**[Course Continuity / Codebase Reference]** — `aurora.asr` does not exist. The actual source tree under `src/aurora/` contains `audio/`, `speech/`, `llm/`, `music/`, `tts/`, `rag/`, `realtime/`, and others — but no `asr/` sub-package. Consequently, `asr_error_analysis.py` is a phantom file. The "本课收束" claim that "`wer_decompose(ref, hyp)` 直接对应 `aurora.asr` 评估脚本中 `asr_error_analysis.py` 的核心逻辑" is doubly wrong: the module doesn't exist, and the function is named `alignment`, not `wer_decompose`. Fix: Either (a) create `src/aurora/speech/error_analysis.py` with a real `wer_decompose` that returns S/D/I counts, and update the reference accordingly, or (b) correct the prose to reference the actual `alignment` function and remove the phantom module path.

### 🟠 Major

**[Exercise Design / Learning Objectives]** — The learning objective "统计最高频的错误模式混淆矩阵（confusion matrix）" promises a confusion matrix, but the implementation delivers only `Counter` frequency lists — a flat ranked list of substitution pairs, not a matrix. A true confusion matrix would be a 2-D structure (ref vocabulary × hyp vocabulary) and would enable visualization of systematic phoneme or lexical confusions. Fix: Either build a genuine confusion matrix (even a small one from the PAIRS vocabulary), or rewrite the learning objective to say "统计最高频的错误对（substitution pairs）" to match what the code actually does.

**[Accuracy / Pedagogical Quality]** — `alignment()` is case-sensitive but `wer()` lowercases both inputs before computing edit distance. For the current PAIRS (all lowercase), this causes no visible difference, but it creates a latent inconsistency: running `alignment("The cat", "the cat")` would report a substitution while `wer("The cat", "the cat")` returns 0.0. A student who tests the functions with real-world mixed-case output will get contradictory results. Fix: Add `.lower()` normalization inside `alignment` (or add an explicit `normalize=True` parameter) and add a short comment noting the design choice.

**[Accuracy]** — The prose in "本课收束" names the wrong function: it says `wer_decompose(ref, hyp)` when the function defined in the notebook is `alignment(ref, hyp)`. The notebook never defines anything called `wer_decompose`, so the summary misrepresents the lesson's API. Fix: Replace `wer_decompose(ref, hyp)` with `alignment(ref, hyp)` in the "本课收束" text, or define a thin `wer_decompose` wrapper that returns `(S, D, I, N)` counts and use that in cell-9.

### 🟡 Minor

**[Runability]** — `import numpy as np` appears in cell-1 but `np` is never referenced in any subsequent code cell. The entire notebook uses only Python built-ins and `collections.Counter`. The dead import is harmless but misleading — at this stage in the course students may assume they need NumPy for this task. Fix: Remove the `import numpy as np` line, or add a brief comment if NumPy is intentionally reserved for a future exercise.

**[Pedagogical Quality]** — Learning objectives are formatted as unordered bullet points. Per course convention (and the review rubric), they should be a numbered list to allow unambiguous cross-referencing ("Objective 2 covers…"). Fix: Convert the three bullet points to `1.`, `2.`, `3.` numbering.

**[Pedagogical Quality]** — Cell-9 computes a single corpus-level WER `(total_S + total_D + total_I) / total_ref_words` without explaining how this differs from the macro-average of per-sentence WERs. These two quantities can diverge significantly (a single very long sentence dominates the corpus metric). At L74, students deserve a one-sentence note on why corpus WER is typically preferred in benchmarking. Fix: Add a comment or markdown note: "注意：这里计算的是语料级 WER（corpus-level），而非句子 WER 的平均值。语料级 WER 以参考词数为权重，是 ASR 评测的行业标准。"

### 💡 建议

- Consider replacing one of the pre-built PAIRS with a blank `("...", "")` slot for students to insert their own ref/hyp pair and observe the counters update — this transforms the passive demo into a minimal interactive experience without requiring full exercise scaffolding.
- The markdown error-type table is genuinely good teaching material. Consider adding a "频率" column showing actual counts from the PAIRS run, tying the table concretely to the code output.
- A simple `matplotlib` heatmap of the substitution counter (ref_word × hyp_word) would fulfill the "confusion matrix" learning objective with only ~10 lines and make the lesson significantly more visual, consistent with the course's laviz/aviz aesthetic.

2. **Fix the phantom `aurora.asr` / `wer_decompose` reference in "本课收束"** — this creates a false impression of codebase integration and will confuse any student who tries to follow the reference.
