## 概述

L04 covers the sinusoid formula `A·sin(2πft+φ)` clearly and the pedagogical flow — from parameter table through a TODO exercise to chord synthesis — is logical and appropriate for this point in the course. However, the notebook has a critical runability fault: the chord-synthesis demo cell unconditionally calls `sinusoid()` before the student implements it, crashing top-to-bottom execution. The "Aurora 连接" claim is also factually wrong in a way that will confuse students who go looking at the actual codebase.

## 亮点

- The parameter table (频率/振幅/相位 → 物理意义 → 谱图效果) is concise and well-structured; it gives students a mental model before seeing any code.
- The reference input/output example (`A=2, f=1, phi=0, t=[0,0.25,0.5,0.75]` → `[0,2,0,-2]`) is an excellent scaffolding device — students can mentally verify it without running code.
- The reasoning-hint block in cell `17f744ee` walks through three explicit steps (angle = 2πft, add phi, multiply by A), which is exactly the level of granularity an early learner needs.
- The chord demo is a strong motivator: it makes the "sine waves are atoms of sound" claim viscerally concrete before DFT is introduced.

## 问题列表

### 🔴 Critical

**[Runability]** — The chord demo cell (`c_chord_demo`) calls `sinusoid()` unconditionally, but the stub returns `None`. Running the notebook top-to-bottom produces `TypeError: unsupported operand type(s) for +: 'NoneType' and 'NoneType'` at `c4 + e4 + g4`, killing every cell from the chord section onward. Students who haven't yet filled in the TODO get a cryptic crash rather than a useful nudge.

Fix: Guard the chord cell with a check, e.g.:
```python
if sinusoid(np.array([0.0]), 1.0, 1.0) is not None:
    # run chord demo
else:
    print("先完成上方 sinusoid() 练习再运行本格。")
```
Or place the chord cell after a completed solution cell and note it requires the implementation.

### 🟠 Major

**[Accuracy / Aurora Connection]** — The notebook states "aurora.audio.sine 就是这节课的核心函数" twice (intro and cell `49b26b19`). In reality `aurora.audio.sine` (defined in `src/aurora/audio/io.py`) has the signature `sine(freq, duration, sample_rate, amplitude)` — it takes no `phi` argument and does not accept a pre-built time array `t`. The lesson's `sinusoid(t, A, f, phi)` is a related but distinct interface. A student who opens `io.py` looking for the `phi` parameter will be confused and may conclude the codebase is inconsistent.

Fix: Revise to: "aurora.audio.sine 是同一个公式的生产版本，但接口不同（它接受 `freq, duration, sample_rate` 而非时间数组 `t`，且无 `phi` 参数）；你将在 L33 实现完全对齐的 `my_sine`。"

**[Exercise Design]** — The stub `return None` instead of `raise NotImplementedError()` produces a confusing cascade: the assert cell fails with a numpy broadcast error (`TypeError` or `ValueError: arrays must be same shape`) rather than signaling that the function is unimplemented. Students see a numpy traceback unrelated to their actual error.

Fix: Change the stub to `raise NotImplementedError("TODO: 实现 sinusoid")`. This also matches CLAUDE.md's implicit convention.

### 🟡 Minor

**[Exercise Design — Test Coverage]** — The single assert only tests `phi=0`. The lesson's stated goal is to teach all three parameters, but there is no test verifying `phi` behavior. A student who accidentally hardcodes `phi=0` would pass all tests.

Fix: Add at minimum one phi test:
```python
y_cos = sinusoid(t, 1.0, 1.0, np.pi/2)
assert np.allclose(y_cos, np.cos(2*np.pi*t), atol=1e-10), 'phi=π/2 时应等于 cos'
```

**[Accuracy — Misleading Claim]** — Cell `cell-7` markdown says "freq 加倍时相邻样本变化幅度翻倍". This is only true for the maximum slope of the wave (the derivative scales with frequency), not for every pair of adjacent samples. At a sample that happens to be near a peak or trough, the change is close to zero regardless of frequency.

Fix: Revise to "freq 加倍时，波形变化的最大速率翻倍（最大斜率 = 2πfA）——对应听感上的音高升高一倍。"

**[Runability — Import Hygiene]** — Cell `cell-15` and the chord demo cell `c_chord_demo` both use `np` without a local import. They rely on `np` being in the kernel namespace from an earlier cell. This is fine in sequential execution but breaks if cells are run in isolation or reordered.

Fix: Add `import numpy as np` at the top of any cell that uses it and runs independently (the chord cell in particular, since it also has its own `plt` import already).

### 💡 建议

1. **Edge-case tests for robustness**: Add `A=0` (should return all zeros) and `f=0` (should return constant `A·sin(phi)`) as assert cases. These catch common implementation mistakes like forgetting the `A` factor.
2. **Euler preview placement**: Cells `cell-5` and `cell-6` introduce `np.exp(1j * angles)` before complex numbers have been taught. The forward link is appreciated, but consider adding a one-line disclaimer like "（复数 `1j` 将在 L05 详解，这里只借用它预览 sin/cos 的来源）" to prevent students from trying to understand the notation before L05.
3. **Chord normalization comment**: The chord cell divides by 3.0 to avoid clipping. A one-sentence explanation ("除以 3 防止三个振幅叠加后超出 [-1,1] 范围") would turn a silent magic number into a teaching moment about headroom.
